Reference Architecture

Three-Layer Scalable Agentic Runtime

A reference architecture for production AI agents — A = ⟨S, H, X⟩ plus an out-of-stack data subsystem 𝒟. The core claim: an agent’s logical, physical and data scaling can be decoupled, and the decoupling point is the explicit contract layer (the Harness).

This page is a first-time reader’s tour. It walks through the three layers, what each layer actually contains, and how the agent loop runs — then states the falsifiable claims that make this a research position, not just an engineering diagram.

1The Three Layers — what each one is for

An agent is built from three stacked layers. Skills say what to do (the task playbook). Harness decides how the work gets done (it turns intent into runnable steps, and owns memory, context, tools, code generation, safety and audit). Scaffold is where it runs safely (an isolated sandbox with shell, network, storage and model-serving).

The trick is that each kind of growth is added on a different axis: add a Skill and the agent can do one more kind of task (logical scaling); add a Scaffold instance and you get more throughput (physical scaling). Because they only ever meet at the Harness contract, they don’t slow each other down. External data grows on yet another axis through the side subsystem 𝒟.

2What lives in each layer — the spec lists

Every layer is made of concrete, buildable pieces. We deliberately ground them in formats that already exist in the wild, so the architecture is an extension layer rather than a new protocol: Skills use the Anthropic SKILL.md format; the Harness capability object is grounded in MCP tool schemas and Claude-Code-style tools; the Scaffold spec covers isolation plus enterprise SSO and cloud integration.

Two things are worth pointing out for builders. First, code generation (Read/Edit/Bash + test/lint) is distinct from tool synthesis (promote validated code into a reusable capsule). Second, system skills (memory compaction, write-back, reflection, retrieval, GC) live inside the Harness — they maintain the runtime, so they are not user-facing Skills.

3How the agent works — a step-by-step walkthrough

The Harness runs a loop. The LLM makes the choices — it plans, picks the next tool (O1), selects the model for the step (O7) and allocates the token budget (O8) — while typed contracts, a policy gate and a verifier keep it safe and on track. This is what we call a probabilistic control plane (claim N0): the decisions are LLM-driven, so they must be constrained by audit logs and invariants.

Each step is checked before it runs, executed on the Scaffold or sent as a query to the data subsystem, then scored by a verifier/reward. If the score is good enough the artifact is delivered; otherwise the agent reflects, revises the plan and loops back. Quietly in the background, system skills keep memory compact and relevant between steps.

4Code as the contract — the real engine of the Harness

Under the hood the Harness is not just a bag of tools; it is a managed runtime whose contract is code. From a Skill’s requirements the LLM drafts a plan — sub-goals and their order, written in natural language and governed by a PRD (the skill’s acceptance spec). For each sub-goal several tools may apply, so the LLM does a probabilistic fan-out: it writes orchestration code for competing candidates, dispatches them to sub-agents (whose dependencies are themselves expressed in that generated code), then verifies and selects the best. That is the agent loop.

Crucially, this plan is not a preset workflow (as in dify / n8n or template-driven agents) — no fixed flowchart, no tools bound to nodes at design time. It is a proactive, LLM-driven agent: at runtime the LLM selects tools from the harness library (T1–T8), reflects and re-composes them, can build several candidate combinations at once, and — when a capability is missing — writes a new tool (T8) to formalize how that combination runs. The plan is a runtime artifact, not a design-time configuration.

This resolves an apparent contradiction. Recent systems argue the scheduler should be pulled out of the LLM into a deterministic kernel. Our answer (N6): the LLM authors the code probabilistically, and the runtime executes it deterministically — the generated, verified code is that deterministic kernel. A successful, reward-gated loop is then crystallized as a Skill (N7): its tool calls and result-causality are frozen into a code-loop that can be shared across users and invoked as a single integrated tool. Because Skill and Harness meet only through this code-loop, each can evolve on its own (N8 / P10): Skills add stronger tools; the Harness absorbs successful skills and rides ever-stronger reasoning models.

5Eight kinds of tool — one capsule interface

A Harness “tool” is not one thing. We classify them into eight kinds that all expose the same intent + I/O contract to the LLM, differing only in implementation and governance: T1 standard tools (web search); T2 model tools (a fine-tuned OCR / classifier called as a tool); T3 crystallized skills (reusable, skill-verified code-loops — N7); T4 data-theme queries (external: TianYanCha / GitHub / arXiv; internal: roster, org chart, quota, territory); T5 harness meta-tools the LLM can call on itself (compact, recap, writeback); T6 scaffold commands (bash, file); T7 human-in-the-loop (ask / confirm / approve — the human as a callable capability); T8 ephemeral code tools the LLM writes on the fly (O5), not yet crystallized.

Genuine additions over the usual tool enum: T2 model-as-tool (a served small model is a first-class capability), T5 (maintenance as an LLM-invoked meta-tool), and the T8 → T3 lifecycle (an on-the-fly generated tool becomes a reusable skill after verify + reward-gate + generalization). T7 makes human-in-the-loop a first-class, auditable tool — and every human confirmation feeds a T3 skill as a positive example. Deriving a sub-agent is not a separate kind: launching one is a Scaffold interface (a fork that claims new CPU / isolation — physical scaling +X); what it runs is either existing tools or a freshly written T8 program; and the N6 code-loop dispatches them.

6Not a preset workflow — a proactive, LLM-driven agent

It is worth stating what this architecture is not. In tools like dify / n8n or template-driven agents, the “plan” is a design-time configuration: a fixed flowchart whose nodes have tools bound to them; to change behavior a human edits the graph.

Here the plan is a runtime artifact. Given a goal in natural language, the LLM selects tools from the harness library (T1–T8), reflects and re-composes them, builds several candidate combinations at once, and — when a capability is missing — writes a new tool (T8) to formalize how the combination runs. The result is orchestration code specific to this run (N6): probabilistic to author, deterministic and auditable to execute. That is the line between a preset agent and a proactive agent.

★The falsifiable core

C5

Logical and physical scaling are orthogonal, with the Harness contract as the decoupling point.

P1

Adding Skills doesn’t drag down throughput (∂Θ/∂|S|≈0); adding Scaffold doesn’t change coverage (∂𝒯/∂|X|=0).

N6

Code is the contract: the LLM authors orchestration code probabilistically; the runtime executes it deterministically & auditably.

P10

Skill ⊥ Harness — coupled only through the code-loop, so each side iterates independently (co-evolution stays at the interface).

N0

The control plane is probabilistic (LLM-driven) and is made trustworthy by audit logs + invariants.

P3

Adding a Skill must preserve safety invariants — benign-in-isolation skills can be harmful in composition.
