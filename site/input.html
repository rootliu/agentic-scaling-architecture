架构总构想 · 2026-06-17

我阅读了论文，并进行了一些离线实验，总结了一个 agentic runtime 的参考架构。一个 agent 的目标是以基础大模型为智力和调度基础的，就解决一个具体 use case 的应用。这个应用有颗粒度不同，即解决不同的行业、企业、角色或者个人的类问题。应用需要有扩展能力，包括逻辑扩展和物理扩展，逻辑扩展是增加新的 specs，物理扩展是增加新的处理能力，throughput。我初步认为自底向上应该分为 3 层，最底下一层是 Scaffold，对应 microVM 等运行沙盒架构；中间一层是 harness，对应 memory、system prompt、context manage 等内部 agent 工具，以及利用 Scaffold 层面的工具（例如 bash）、还有外部工具（例如多模态的视频感知、搜索等），并具有大模型调用产生新的程序工具的能力。最上层是 skills，包括 specs 的描述、输出格式、工具调用条件、输出条件、循环条件等，目标是组装和迭代推送到大模型的 context。这个体系对应互联网部署里面的虚机、软件框架运行时和应用前端运行时。这样，agent 可以通过调用 skills 来实现逻辑扩展，通过增加 Scaffold 来实现物理扩展，增加能力细化过程以及 throughput。通过针对不同层的性能、可靠性、可用性、安全等非功能需求的设计和优化，这个 agent 可以具备不同行业和业务的服务条件，可以相对稳定运营，起到数字员工的作用。

维护子系统走物理扩展 · 2026-06-18

这种 harness 层的 sub agent，对外的接口是相对固定的，相当于一个函数 tool。但是由于模型能力不断提升和优化，从而使得其实现需要依赖于 LLM 侧的能力。而且这压缩一类的功能确实有模型调用的能力，但这个模型可能会和 agent 使用的不一致，其 token 消耗相对是线性的，例如每到 32K 就压缩、每到 128K 就写入 memory，这些可以用 haiku 这种模型来干，不需要用 opus 级别的。而给模型的指令是一致的，所以相关的操作会随着模型优化不断优化，输出的 token 也会得到优化，所以这个过程支持线性扩展。基于此，这个 harness sub agent 应该也算物理扩展部分。
