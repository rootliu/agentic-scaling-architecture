Scalable Agentic Runtime, and the research problem of making it manageable.

Reading order: start from the S/H/X runtime core and the Harness contract gate, then read the CP/DP split horizontally. The 𝒟 subsystem sits outside the stack; P10-P13 explain parallelism/locality; N8/P14 hardens validated skills into code as the deterministic anchor of a probabilistic control plane.

This project studies how an agentic runtime can keep auditability, governance, operability and evolution under control while continuously adding skills, execution resources, external data sources and parallel sub-agents. Extension axis, control/data plane, out-of-stack data subsystem, and parallelism/locality are four orthogonal cuts; Skill-as-Code provides the deterministic anchor.

unified proposition · meta-claim

For each of the four cuts, the first-order cost of “adding more” lands in a plane or subsystem decoupled from the main request path: logical growth lands in the control plane, physical growth in the data plane, data growth in an offline off-policy loop, and parallel growth in tool-closure/locality planning. The goal is an agentic runtime that is scalable and manageable.
core novelty = C5 logical scaling = Skills physical scaling = Scaffold data subsystem = 𝒟 (out-of-stack) parallelism/locality = P10-P13 deterministic anchor = Skill-as-Code

Three-layer stack ⟨S · H · X⟩ + out-of-stack data subsystem 𝒟

Three planes, one contract;
plus a data subsystem.

The stack separates responsibilities bottom-up, analogous to infrastructure / framework runtime / front-end runtime. Outside the three layers sits a semi-independent data subsystem; it is not a fourth layer.

L3 · Logical axis

Skills

Specification Plane. Specs, I/O schemas, call / termination / loop conditions. Its job is to assemble and iteratively push task-relevant material into the LLM context. Task skills live here.

 L2 · Contract

Harness

Capability / Contract Plane. It translates upper-layer intent into lower-layer executable units. It has three parts: contract translation, capability supply through tool synthesis, and the manageability subsystem M.

 L1 · Physical axis

Scaffold

Execution / Isolation Plane. microVMs, sandboxes, isolation and compute. In the broad definition it also includes inference serving capacity and characterizes system throughput Θ.
Out-of-stack · data subsystem 𝒟

Data Subsystem (not a fourth layer)

It describes how agents understand, access, govern and maintain authorized external data. It splits along online/offline and CP/DP: D₁ retrieval sits in the data plane; D₂ semantic summarization is an out-of-stack off-policy loop 𝓛₂ with its own loop, sub-agents and model; D₃ governance and D₄ lifetime sit in the control plane. Only the semantic-join query interface enters the main path.
Core argument
Modern cloud applications already scale front ends horizontally by traffic, business logic vertically by capability, and back ends by compute capacity. Multi-axis decoupling is a known engineering paradigm. The delta in this research is to define the necessary and sufficient condition (the Harness contract layer), falsifiable criteria (P1-P14), and a shared invariant across the four orthogonal cuts under probabilistic LLM calls: high-frequency online paths are separated from low-frequency, offline or plannable paths.

This reframes the contribution from “we propose a three-layer architecture” to “we show how logical, physical, data and parallel scalability can be decoupled under LLM uncertainty, and how they reuse the same decoupling invariant; stable skills can also be hardened into code to reduce model-version drift.” That is the answer to the “engineering diagram vs. research claim” challenge.

Novelty map four cuts

C5 + N0-N8, in one table.

C5 is the core. N0-N8 expand it through control plane, data subsystem, parallel locality and Skill-as-Code. Each point is tied to a falsifiable proposition or an experimental design.

C5 · Extension axis

Orthogonal logical/physical scaling

Logical and physical scaling are orthogonal, and the decoupling point is the Harness contract layer. None of the seven 2026 papers split the two axes and bind them to different layers. → P1-P3

 N0 · Plane

Probabilistic control plane

LLM-driven control decisions, including contract translation, reflection and tool synthesis, require audit logs plus invariant constraints. This differs from deterministic SDN / Kubernetes control planes. → P4-P6

 N1 · Data

Off-policy semantic summarization

A schema-on-read background loop summarizes into an untruncated lake, while semantic joins inject only what is needed. It fills the coverage/precision gap left by Halevy (2605.28787). → P8

 N2 · Data

Data governance as skill

Governance moves from boundary control to experience accumulation through per-use-case data-usage skills. This stitches data governance to agent skill and memory theory. → P9

 N4 · Data structure

Independent out-of-stack off-policy loop

Semantic summarization is not an online step or interface; it is an offline subsystem orthogonal to the online runtime, with its own loop, sub-agents and model, similar to database vacuum or offline indexing.

 N3 · N5

Architectural consistency + skill triad

N3: the data subsystem naturally splits along CP/DP and reuses the same plane invariant. N5: skills divide into task, system and data-usage skills under one skill form.

 N6 · Parallel locality

Sub-agents by tool-set partition

Tool-set closure defines parallel independence, Skills locality and Scaffold seed boundaries, making parallelism a designed runtime property instead of a scheduling parameter. → P10-P12

 N7 · Dry Run

Planning-time parallelism probe

Before execution, dry-run probes tool-set closure and environment scope, then partitions sub-agents to reduce cross-agent overlap ρ. → P13

 N8 · Skill-as-Code

Deterministic anchor for a probabilistic CP

A validated skill with a closed operation space can be hardened into code, decoupling its output from model version θ and improving auditability, reproducibility and contract stability. → P14

Dry Run · instantiated walkthrough four-dimensional alignment

The abstract proposition, grounded in one run.

Concrete tasks walk through the full A=⟨S,H,X⟩ + 𝒟 stack, showing both the spec invocation order and the data flow. Since v0.2, the fourth dimension is explicit: the planning phase first uses dry-run to probe tool-set closure and Scaffold environment (P13), partitions parallel sub-agents by non-overlapping tool sets (M1/M2), freezes each stable prefix (P11), derives execution by seed affinity (M4), and passes only summarized context (M3) across sub-agents.

Use case U · AI4Science

Perovskite band-gap screening (~1.3 eV)

This case exercises the full stack plus data subsystem: papers feed D₂ temporal summaries, property databases feed D₁ retrieval, simulations trigger code generation plus Scaffold cloud/HPC, multi-signal validation gates the result, and reports/write-back update D₂/Ω/D₃. Sub-goals {literature → candidates → properties → screening → analysis → report} are partitioned into parallel sub-agents through dry-run.

P7 retrieval decoupling · P2 contract translation · P10-P13 parallelism/locality · §4.4.1 anti-hallucination

 Use case K · FinTech

Kronos foundation model for stock prediction

An end-to-end time-series skill: D₁ data retrieval, GPU-hosted pretrained model as Scaffold serving, probabilistic sampling with T/top_p/sample_count in the loop, multi-signal validation through backtest/leakage/calibration, and predictions written back as expiring records. Multiple tickers and horizons are parallelized by tool-set partition.

P1 physical scaling · P3 capability fence (read-only) · P10-P13 multi-ticker parallelism · N0 auditable sampling
How the four dimensions appear in one run

The same spec invocation sequence leaves first-order traces along all four orthogonal cuts. That is the runtime evidence for the shared decoupling invariant:

Extension axis A Skill only declares required_capabilities; H translates 𝓘→𝓔 and never binds directly to Scaffold (P2).

CP/DP planning / routing / verification stay in CP; retrieval / compute / serving stay in DP (P4-P6).

Data subsystem the runtime checks D₃/D₄/D₂ shared summaries before raw retrieval, so adding sources does not raise single-request cost (P7); write-back is append-only and traceable to evidence.

Parallelism/locality dry-run probes tool-set closure and partitions by non-overlap (P13/M1/M2); same seeds co-locate (M4), prefixes freeze (P11), and only summaries cross boundaries (M3) → throughput approaches 1/s (P10).

First-principles mechanism (§0.5): one inference equals intrinsic knowledge (fixed weights) plus context (variable). The agent can only organize context. Frozen prefix + changing suffix buys cache reuse, reduced attention dilution and parallel independence at the same time; P10-P13 are four falsifiable projections of that root cause.
