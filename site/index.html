逻辑与物理，
本可解耦。

一个 agentic runtime 的三层参考架构 —— Scaffold · Harness · Skills。 我们主张：agent 的逻辑可扩展性与物理可扩展性可以正交解耦，解耦点是层间的显式契约。

核心 novelty = C5 逻辑扩展 = Skills 物理扩展 = Scaffold
查看架构 → 文献证据
三层栈

三个平面，一条契约。

自底向上，每一层各司其职，对应互联网部署的"虚机 / 框架运行时 / 前端运行时"。

L3 · 逻辑轴

Skills

Specification Plane。specs 描述、I/O schema、调用 / 终止 / 循环条件。职责是组装并迭代推送进 LLM context。仅含 task skills。

L2 · 契约

Harness

Capability / Contract Plane。把上层意图翻译为下层可执行单元。三分：契约翻译 + 能力供给（tool synthesis）+ 维护子系统。

L1 · 物理轴

Scaffold

Execution / Isolation Plane。microVM / sandbox、bash、隔离与算力。宽定义含推理 serving 容量，刻画系统 throughput。

核心论证（修订版）
现代云应用早已"前端按流量横扩 / 业务按功能纵展 / 后端按算力扩"——三轴解耦是已知工程范式。 本研究的 delta：在每层内嵌概率性 LLM 调用的条件下，给出该解耦成立的充要条件（契约层 H）与可证伪判据（P1）。

这把论文主张从"我们提出三层架构"提升为"我们证明在 LLM 不确定性下双扩展维度可以解耦"——回应"工程而非科学"的核心质疑。

最新精化 2026-06-18

维护子系统，走物理轴。

system skills（压缩 / 写回 / 反思）功能属 Harness，但扩展性属物理轴 —— 正如 SaaS 的 SSR 前端。

Skills 二分

task skills 完成用户任务（L3）；system skills 维护 runtime 自身，归入 Harness。判据：输出是否进入用户可见结果 + 触发源。

触发 ∝ token 流量

每 32K 压缩、每 128K 写 memory，用 Haiku 级廉价模型、指令固定 → 支持线性扩展，故归物理扩展。

P1 机制解释

维护子系统吸收"逻辑扩展带来的 context 膨胀"，这正是"加 task skill 不拖垮 throughput"的原因。
