这正是"共享同一解耦不变量"的运行时证据。机制根因见 §0.5 第一性原理。

扩展轴

Skill 只声明 required_capabilities，H 翻译 𝓘→𝓔，从不直接绑 Scaffold（P2）。

控制/数据面

plan / 路由 / 验证在 CP，取数 / 计算 / serving 在 DP（P4–P6）。

数据子系统

先查 D₃/D₄/D₂ 共享摘要再取原文，源增加不抬单请求成本（P7）；写回逻辑追加、可回指原文防幻觉。

并行/局部性

dry-run 探工具集闭包再按不重合切分（P13/M1/M2），同 seed 共置（M4）、前缀冻结（P11）、只传摘要（M3）→ 吞吐逼近 1/s（P10）。

机制根因：一次推理 = 内在知识（权重，固定）⊕ context（可变），agent 唯一能动的是组织 context；前缀冻结 + 尾部变化同时买到缓存复用（快）、注意力不稀释（准）、并行独立（可切分）——P10–P13 即此根因的四个可证伪投影。
用例 U · AI4Science

钙钛矿带隙筛选（~1.3 eV）

Spec 调用顺序（U）：规划期先 P13 dry-run 探工具集闭包、按不重合切成三束并行子 agent（M1/M2）；Scaffold 依 seed 亲和派生（M4），各子 agent 冻结稳定前缀 → H↑（P11），彼此只沿摘要松耦合传递（M3）；回写逻辑追加、带 evidence_ref 防幻觉。

四条结构性约束（对应命题）

· [06–08] 先于 [14]：先查 D3/D4/D2，命中共享摘要就不重拉原文 → 减少多 agent 争用、降单请求成本（P7）。

· [10–13] 经 Harness 契约：Skill 只声明 required_capabilities，由 H 翻译 𝓘→𝓔（P2）。

· [02b–02c/13b] 规划期 dry-run 切分 + seed 亲和：先探测工具集闭包/环境范围（P13/N7）再按工具集不重合切并行子 agent（M1/M2），同工具集同 seed 共置（M4）→ 重合度 ρ↓、前缀命中 H↑、并行度 k_max↑，吞吐逼近 1/s（P10）。稳定工具集/记忆入前缀并冻结（P11），跨子 agent 仅传摘要（M3）。

· [21] 逻辑追加 + evidence_ref：溢出降冷不硬删，旧结论可回指原文，防幻觉（§4.4.1 S3/S4）。

Data Diagram（数据流，在线/离线双路）

数据流（U）：离线 𝓛₂ 把原文预消化成带时序约束的 SummaryEntry 存入共享只读摘要 DB（不在主请求路径 → 新增源不抬单请求成本，P7）；在线只做 D₂.semantic_join + D₁ 取数，均经 Harness 契约；执行结果经 Ω 蒸馏回灌 D₂/D₃，逻辑追加、可回指原文防幻觉。
缩写：arXiv=学术预印本库 · MP=Materials Project（开放 DFT 材料物性库）· OQMD=Open Quantum Materials Database · DFT=密度泛函理论（第一性原理算物性）· HPC=高性能计算集群 · microVM=微虚拟机沙箱 · SSO=单点登录

  1. 两条数据路径分离：离线 𝓛₂ 把原始文献/库预消化成带时序约束 SummaryEntry，不在主请求路径上 → 新增源不抬单请求 token 成本（P7）。
  2. 摘要 DB 是共享只读热点：多 agent 读同一份 SummaryEntry(快照读)，不各自重拉同一篇 PDF → 消除争用、省重复推理。
  3. 写回单向、逻辑追加：执行结果 → Ω → 蒸馏新 SummaryEntry 回灌 D2 + 更新 D3，每个可召回结论都能回指原文，防幻觉。
  4. 并行切分是一层正交结构：Harness 在主路径之前用 P13 dry-run 探得每个 sub-goal 的工具集闭包，据此按工具集不重合切成并行子 agent；子 agent 各自读同一份共享摘要 DB（前缀可冻结、H↑），彼此只沿摘要接口松耦合汇聚 → 串行比 s↓、吞吐逼近 1/s（P10）。

Spec 调用顺序集

Spec 调用顺序（K）：多标的/多周期在规划期 P13 dry-run 探得共享同一 kronos_predict 工具集闭包，可同 seed 共置并行（M1/M2/M4），前缀冻结 → H↑、k_max↑；预测作为带有效期 ForecastEntry 回写，过期不静默复用；策略上只读、泄漏防护经契约（P3）。

结构性约束（↔ 命题）

· 预测 = 带时序约束的记录：ForecastEntry 仅在 [valid_from, valid_to] 内有效；过期窗口逻辑失效、不静默复用 → 防陈旧信号幻觉（§4.4.1 S3）。

· 泄漏防护经契约：y_timestamp 严格晚于 x 窗口；verifier 强制；Skill 不直接拿原始未来 K 线（P2/P3）。

· 概率采样在循环内且可审计：T/top_p/sample_count 记入 RunTrace；同输入+种子可复现（N0）。

· 策略上只读：不授予下单能力；policy gate 对任何交易通道工具 fail-closed（P3 能力围栏）。

· 多标的并行按工具集切分：规划期 dry-run 探得各标的子任务共享同一 kronos_predict 工具集闭包，可同 seed 共置并行（M1/M2/M4），跨子 agent 仅传预测摘要（M3）→ 吞吐逼近 1/s（P10/P13）。

Data Diagram（数据流，预训练模型托管 + 带有效期预测）

数据流（K）：预训练 Kronos 模型作为 Scaffold 上 GPU 托管 capsule（加标的=加 serving，不动 Skill，P1）；在线经 AKShare(D₁) 取 OHLCV，预测结果作为带有效期 ForecastEntry 落入 D₂——仅窗口内有效、过期重算；验证是多信号（泄漏检查 + 回测 + 校准），策略上只读（P3）。
缩写：AKShare=开源金融数据接口库 · OHLCV=开/高/低/收/量（K 线五字段）· Kronos=时序基础模型 · GPU=推理加速卡 · microVM=微虚拟机沙箱 · SSO=单点登录 · Sharpe=夏普比率

  1. 预训练模型作为被托管的能力：kronos_predict 是 Scaffold 上 GPU 托管 capsule；增加标的在物理轴扩(加 serving)，不动预测 Skill（P1）。
  2. 预测是带有效期的数据：以 ForecastEntry 落入 D2，仅窗口内有效；后续运行仅在仍有效时复用，否则重算。
  3. 验证是多信号：泄漏检查 + 回测指标 + 校准度，而非单一准确率（H9）。
  4. 并行度可设计：多标的/多周期子 agent 工具集不重合、同 seed 共置，前缀冻结（kronos capsule 入前缀）→ H↑、k_max↑，吞吐逼近 1/s（P10–P13）。
