# 可扩展 Agentic Runtime 与可管理性研究简报

> 配套材料：`architecture.html` · `references.html` · `timeline.html` · `input.html`

## 0. 研究定位

本项目的目标是研究一个**可扩展的 agentic runtime**，以及它在规模化之后的**可管理性**：当 agent 系统持续增加 skills、工具、执行资源和外部数据源时，runtime 如何仍然保持可审计、可治理、可运维、可演进，并且让新增能力的一阶成本不直接压到主请求路径上。

## 1. 一句话概述

本项目提出一个面向 agentic runtime 的三层参考架构：`A = <S, H, X>`，其中 Skills 负责逻辑能力表达，Harness 负责契约翻译与能力供给，Scaffold 负责执行、隔离和吞吐。研究的核心不只是提出三层栈，而是解释和验证：在 LLM 调用具有概率性的条件下，agent 的逻辑可扩展性、物理可扩展性和数据可扩展性如何通过显式契约与异步子系统实现正交解耦，并进一步形成可审计、可治理、可维护的 runtime 管理面。

## 2. 要解决的问题

当前 agent 系统常把三类扩展混在一起处理：

- 逻辑扩展：新增任务、技能、工具和业务能力。
- 物理扩展：提升并发、吞吐、隔离强度和服务容量。
- 数据扩展：接入更多外部数据源，并让 agent 正确理解、访问和治理这些数据。

这三类扩展一旦耦合，系统会出现典型问题：新增 skill 拖垮在线吞吐，新增 sandbox 不提升真实任务覆盖，新增数据源显著增加单次请求 token 成本，治理逻辑散落在提示词、工具和权限系统中，难以审计和复用。

本研究试图回答一个更基础的问题：在 LLM 参与决策、工具生成和维护的情况下，agent runtime 是否仍然可以像云系统一样实现“功能扩展”和“算力扩展”的解耦，并进一步把外部数据扩展也纳入同一套解释框架。换句话说，研究对象不是“如何多接几个工具”，而是“agentic runtime 如何在持续扩展时仍然可被管理”。

## 3. 核心架构

### 三层栈 `A = <S, H, X>`

| 层 | 名称 | 主要职责 | 扩展含义 |
|---|---|---|---|
| L3 | Skills | task specs、I/O schema、调用/终止/循环条件，组装进 context | 逻辑扩展，提升能力覆盖 `T` |
| L2 | Harness | 契约翻译、能力供给、工具路由、维护子系统 | 解耦点，把意图 `I` 转为可执行单元 `E` |
| L1 | Scaffold | microVM、sandbox、serving、隔离与算力 | 物理扩展，提升吞吐 `Theta` |

其中 Harness 是关键。Skills 不直接依赖 Scaffold，Scaffold 也不理解 Skills 的语义复杂度。二者通过 Harness 的显式契约交互，从而使逻辑扩展与物理扩展可以分别优化。

### 三条正交切分

项目把架构扩展为三条互相正交的切分：

1. 扩展轴：逻辑扩展通过增加 Skills 完成，物理扩展通过增加 Scaffold 容量完成。
2. 控制面/数据面：控制面负责决策、契约翻译、路由、维护触发和不变量；数据面负责 token path、工具执行、sandbox 计算和 serving。
3. 数据子系统：三层栈之外挂接 `D = <D1, D2, D3, D4, Omega>`，处理取数、语义总结、治理记忆和数据 lifetime。

统一的不变量是：新增能力的一阶成本，应当落在与主请求路径解耦的平面或子系统上。逻辑扩展落控制面，物理扩展落数据面，数据扩展落离线 off-policy loop。

## 4. 数据子系统设计

数据子系统不是第四层，而是三层栈之外的协同系统，用来刻画 agent 如何使用外部数据。

| 组成 | 含义 | 作用 |
|---|---|---|
| D1 | on-policy 取数 API | 在运行时访问特定数据源和切片 |
| D2 | off-policy 语义总结平面 | 在主请求之外异步预总结数据源，形成可 semantic join 的摘要 |
| D3 | 数据治理 memory | 把“如何用数据”沉淀为 per-use-case data-usage skill |
| D4 | 数据 lifetime 体系 | 管理时新性、业务口径、访问预算和 NFR |
| Omega | agent-owned data memory | 保存 agent 对数据使用经验的长期记忆 |

这里最重要的是 D2 与 D3：D2 把大规模数据理解从在线路径移到离线循环中，D3 把治理从“边界 enforcement”升级为“经验沉淀”。这样新增数据源不应直接污染单次请求的 token 成本。

## 5. 主要创新点

| 编号 | 创新点 | 说明 |
|---|---|---|
| C5 | 逻辑/物理扩展正交解耦 | 把 agent 可扩展性拆成 Skills 与 Scaffold 两个轴，并用 Harness 作为解耦契约层 |
| N0 | 概率性控制面 | 控制面部分由 LLM 驱动，决策具有不确定性，因此必须引入审计日志和不变量约束 |
| N1 | off-policy schema-on-read 语义总结 | 在后台异步预总结数据源，兼顾在线发现的覆盖率和人工元数据的精度 |
| N2 | 数据治理即 skill | 将数据使用策略沉淀为可复用、可演化的 data-usage skill |
| N3 | 三维复用同一平面不变量 | 扩展轴、控制/数据面、数据子系统共享同一个解耦机制 |
| N4 | 栈外独立数据 loop | 把语义总结识别为独立异步子系统，而非在线步骤或普通接口 |
| N5 | skill 理论三分 | 将 skill 从 task/system 二分扩展为 task/system/data-usage 三分 |

## 6. 可验证命题

研究不是只给出设计图，而是给出可证伪命题：

- P1-P3：验证逻辑扩展与物理扩展是否解耦，以及 Harness 契约是否是充分条件。
- P4-P6：验证控制面与数据面是否分离，维护子系统是否能跨面拆解。
- P7-P9：验证新增数据源是否不提升单请求 token 成本，off-policy 摘要是否兼顾覆盖与精度，治理 memory 是否随用例重复而收敛。

可行的实验路径包括：固定 Scaffold 增加 Skills 观察 throughput，固定 Skills 增加 Scaffold 观察吞吐曲线，移除 Harness 做消融，对比在线数据发现、人工元数据和 off-policy 语义总结三种取数策略。

## 7. 当前定位与下一步

当前项目处在“学术研究 + 架构成型 + 文献对照验证”阶段，目标是形成一篇关于可扩展 agentic runtime 及其可管理性的论文，并沉淀一个可开源的 agent 云系统原型。下一步可以围绕三件事推进：

1. 收紧形式化定义：明确 `S/H/X/D` 的接口、成本函数、约束条件和证伪边界。
2. 做最小原型：实现 Harness 契约层、Scaffold 隔离执行、Skills 注册与数据子系统的简化版本。
3. 设计实验矩阵：围绕 P1-P9 做吞吐、覆盖、token 成本、审计性、治理收敛和运维可控性的验证。
