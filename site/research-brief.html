01 · Summary

一句话概述

本项目不是只提出三层栈，而是研究可扩展 agentic runtime 在规模化之后如何保持可审计、可治理、可运维和可演进。

面向 agentic runtime，定义三层栈 A = <S,H,X>：Skills 负责逻辑能力表达，Harness 负责契约翻译与能力供给，Scaffold 负责执行、隔离和吞吐。核心主张是：在 LLM 调用具有概率性的条件下，逻辑可扩展性、物理可扩展性和数据可扩展性仍可通过显式契约与异步子系统实现正交解耦，并形成可管理的 runtime 控制面。

完整演示可以从 架构总图 进入；这份简报负责给第一次接触项目的人快速建立上下文。

02 · Problem

要解决的问题

当前 agent 系统常把新增能力、扩容和外部数据接入混在同一条在线路径里处理。

逻辑扩展

新增任务与 skill

新增业务能力、工具、任务规格、调用条件和终止条件。风险是 skill 变多后拖垮 context、维护和吞吐。

物理扩展

提升并发与隔离

新增 sandbox、microVM、serving 容量和执行资源。风险是只扩算力却不提升真实任务覆盖。

数据扩展

接入外部数据源

让 agent 理解、访问和治理更多数据。风险是每加一个源都增加单请求 token 成本和治理复杂度。

研究问题：在 LLM 参与决策、工具生成和维护的情况下，agent runtime 是否仍然可以像云系统一样实现功能扩展与算力扩展的解耦，并进一步把外部数据扩展纳入同一套解释框架？换句话说，agentic runtime 如何在持续扩展时仍然可被管理？
03 · Design

核心架构

三层栈 A = <S,H,X> 中，Harness 是显式契约层，也是逻辑与物理解耦的关键。

层名称主要职责扩展含义
L3Skillstask specs、I/O schema、调用/终止/循环条件，组装进 context逻辑扩展，提升能力覆盖 T
L2Harness契约翻译、能力供给、工具路由、维护子系统解耦点，把意图 I 转为可执行单元 E
L1ScaffoldmicroVM、sandbox、serving、隔离与算力物理扩展，提升吞吐 Theta
切分 1

扩展轴

逻辑扩展通过增加 Skills，物理扩展通过增加 Scaffold。二者通过 Harness 契约交互。

切分 2

控制/数据面

控制面负责决策和不变量，数据面负责 token path、工具执行、sandbox 计算和 serving。

切分 3

数据子系统

三层栈之外挂接数据系统，处理取数、语义总结、治理记忆和数据 lifetime。

04 · Data Subsystem

数据子系统不是第四层

它是栈外协同系统，刻画 agent 如何理解、访问、治理和维护被授权的外部数据。

D1

取数 API

on-policy 访问特定数据源和切片。

运行时数据面
D2

语义总结平面

off-policy loop 在主请求之外异步预总结数据源，形成可 semantic join 的摘要。

栈外 loop
D3

治理 memory

把“如何用数据”沉淀为 per-use-case data-usage skill。

控制面 API
D4

lifetime

管理时新性、业务口径、访问预算和 NFR。

控制面 API
最关键的是 D2 与 D3：D2 把大规模数据理解从在线路径移到离线循环中，D3 把治理从“边界 enforcement”升级为“经验沉淀”。因此新增数据源不应直接污染单次请求的 token 成本。
05 · Novelty

主要创新点

C5 是核心贡献；N0-N5 沿控制面、数据子系统和 skill 理论继续展开。

编号创新点说明
C5逻辑/物理扩展正交解耦把 agent 可扩展性拆成 Skills 与 Scaffold 两个轴，并用 Harness 作为契约层
N0概率性控制面控制面部分由 LLM 驱动，必须引入审计日志和不变量约束
N1off-policy schema-on-read 语义总结后台异步预总结数据源，兼顾在线发现覆盖率和人工元数据精度
N2数据治理即 skill将数据使用策略沉淀为可复用、可演化的 data-usage skill
N3三维复用同一平面不变量扩展轴、控制/数据面、数据子系统共享同一个解耦机制
N4栈外独立数据 loop把语义总结识别为独立异步子系统，而非在线步骤或普通接口
N5skill 理论三分将 skill 从 task/system 二分扩展为 task/system/data-usage 三分
06 · Validation

可验证命题

这套架构不是只给出设计图，而是给出一组可证伪承诺。

P1-P3

扩展轴

验证逻辑扩展与物理扩展是否解耦，以及 Harness 契约是否是充分条件。

P4-P6

控制/数据面

验证控制面与数据面是否分离，维护子系统是否能跨面拆解。

P7-P9

数据子系统

验证新增数据源是否不提升单请求 token 成本，治理 memory 是否随重复用例收敛。

可行实验路径：固定 Scaffold 增加 Skills 观察 throughput；固定 Skills 增加 Scaffold 观察吞吐曲线；移除 Harness 做消融；对比在线数据发现、人工元数据和 off-policy 语义总结三种取数策略。
