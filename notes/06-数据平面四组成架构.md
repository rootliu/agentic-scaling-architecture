> **类型**: 论文核心理论扩展（formalization extension）
> **状态**: 草稿 v0.1
> **定位**: 把 [05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md) 里抽象的"数据面 DP"细化为一个**略独立于三层栈的子架构**——agent 如何**理解、访问、治理、维护**其被授权的外部数据。提出四组成（接口①数据访问 / 接口②off-policy 语义总结 / 接口③数据治理 memory / 接口④lifetime 体系），并论证其与 C5（[01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md)）和控制/数据面（[05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md)）的正交关系，给出形式化、命题与实验设计。

## 0. 两个"数据面"的概念消歧（必读）

[05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md) 的"**数据面 DP**"是**运行时执行平面**（token 处理路径，∝ token 流量），与"控制面 CP"对偶。
本文的"**数据平面（Data Plane / 数据架构）**"是**agent 与外部数据交互的子系统**——它本身又横跨 CP/DP：数据的**访问执行**在 DP，数据的**语义/治理/生命周期决策**在 CP。

为避免混淆，下文统一术语：
- 沿用 §05 的执行平面记号：**运行时 DP / 运行时 CP**。
- 本文主体称 **数据子系统 $\mathcal{D}$**（the agent data subsystem），其四组成记为 $\mathcal{D} = \langle D_1, D_2, D_3, D_4\rangle$。

> **核心主张**：$\mathcal{D}$ 的四组成恰好沿运行时 CP/DP 分裂——$D_1, D_2$（访问 + 语义总结）主要落在**运行时数据面**（可由 Harness 直接调用、随 token 流量计费），$D_3, D_4$（治理 memory + lifetime）主要落在**运行时控制面**（独立 API、低频、∝ 状态事件，供 context 组装时查询）。这把 §05 的平面切分**第三次**复用，是本架构论证一致性的强证据。

## 1. 问题：agent 与数据协同工作时的认知缺口

agent 解一个 use case 时，需访问其**被授权**的数据，来源异构：

| 来源类 | 例 | 访问特性 |
|---|---|---|
| 企业内部系统 | ERP / SCM / CRM | 结构化、强权限、强一致 |
| 第三方 SaaS | Amazon 电商数据 | API 受限、配额、外部时延 |
| 个人/部门文件 | Excel / 报告 | 半结构化、无 schema、易截断 |
| 互联网公开 | arXiv / blogs | 非结构化、海量、噪声大 |

访问任一数据，agent 必须理解五类信息（用户 I-6 原话归纳）：

1. **语义**：这部分数据是什么、字段含义、业务口径。
2. **存储与访问方式**：在哪个系统、用什么协议/API/配额访问。
3. **数据周期内容**：时效性、地域性、部门性（同一指标随时间/地区/部门口径不同）。
4. **跨数据语义联系**：供应链上下游、CPO/PCB 之于芯片封装、CPO 之于 AI 网络、飞机与货包……即**领域知识图谱式的关联**。
5. **非功能需求 NFR**：安全（授权范围）、可用性（多 agent 争用同一 DBMS）。

最后，agent 还会**产出检索意图**去发现"新的、未知的数据"（例如用 I-6 原语去检索是否已有类似论文——本任务本身就是一例）。

> **缺口诊断**：传统 ETL/数据湖把上述 1–5 交给**人**预先定义（schema、元数据、口径），再灌入 lake/warehouse。但人定义会**截断**原始信息（schema-on-write 丢弃 latent 字段），且**跟不上** agent 的 use case 速度。需要一个 **agent-native 的数据子系统**。

## 2. 四组成定义

### $D_1$ — 数据访问接口（on-policy data access API）

agent 依 Harness 当下需求，**on-policy** 地访问某个特定数据源的特定切片。
- 例：Amazon 电商，6/1–6/19，3 个特定 ASIN 的**日库存变化**。
- 形式：参数化查询 $q = \langle \text{src}, \text{scope}, \text{time}, \text{proj}\rangle$，返回原始数据 $R(q)$。
- 平面归属：**运行时数据面**（执行即耗 token/带宽，∝ 流量）；**Harness 可直接调用**。

### $D_2$ — off-policy 语义总结平面（off-policy semantic summary plane）

**与传统 ETL 的本质区别**：不由人定义规范/元数据，而由 **off-policy agent 主动**搜索并总结每个数据源的**数据结构、数据内容、可能的元数据（含 latent data）**，写入一个 **off-policy lake**，以**避免人为截断、减少原始信息损失**。

- 产物：每个源 $\text{src}$ 的语义摘要 $\Sigma(\text{src}) = \langle \text{schema}^*, \text{content-profile}, \text{latent-fields}, \text{rels} \rangle$。
- agent 通过 **semantic join** 检索相关摘要 $\Sigma$，然后**二选一**：(a) 经 $D_1$ 拉原始数据；(b) 直接用摘要数据。
- 平面归属：摘要的**生产**离线、低频、off-policy（更近运行时控制面）；摘要的**消费/semantic join** 在 on-policy 路径上（运行时数据面）。**Harness 可直接调用**（查询接口）。

> **off-policy 的双关**：既指"非当前任务策略驱动的后台 agent"（RL 语义），也指"schema-on-read、不预先固化"——两层含义都服务于"**不让人/不让当前任务提前截断数据**"。

### $D_3$ — 数据治理 memory 平面（data-governance memory）

除规范 agent 的数据**使用范围**外，把"agent 经 $D_1/D_2$ 拿到的数据 + 语义摘要 + semantic join 结果"**如何与 use case 结合**、不同 scaling 的 agent **如何扩展**这些逻辑，**记录为 memory**：

- agent 怎样用数据最好；
- semantic join 中 **top-$k$** 数据对结果的影响、**权重**；
- 哪些数据**可被其它数据补充**；
- 哪些数据使用时**需上网搜新源**。

即 **per-use-case 的"数据使用 skills 集合"**。agent 访问 $D_3$ 来**相对固定数据使用流程**，并**反向影响**对 $D_1/D_2$ 两种取数方式的采纳。
- 平面归属：**运行时控制面**（决策/策略、低频、∝ use case 事件）；**独立 API**，供 context 组装时查询。

> **关键洞察**：$D_3$ 把"数据使用经验"沉淀为 **skill**——这与 研究原则-外部记忆即Skill基础（外部记忆须提供完整 SOP 才成 skill）和 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) 的 system skills 二分**完全同构**。$D_3$ 是"**data-usage skill**"，是 task skill 与 system skill 之外的第三类，但仍遵循同一 skill 形式 $\langle\text{desc},\sigma^{in},\sigma^{out},c\rangle$。

### $D_4$ — lifetime 体系（data lifetime / freshness & availability）

- **数据时新性**：哪些数据现在最好、哪些历史数据可用、哪些已不可用、时效性如何影响现有数据。
- **业务主题口径（有时效性的定义）**：例如"销售"以**回款**还是**合同**为准——口径随时间/政策变。
- **访问 NFR**：并发 throughput、response time、volume、访问途径——**防止空访问与阻塞**等降低可用性的行为（呼应 I-6 的 DBMS 争用）。
- 平面归属：**运行时控制面**（策略与约束，低频）；**独立 API**，供 context 组装时查询。

> **闭环**：四组成 + 一个"放 agent 自有数据的 memory 体系"（即 [03-Harness-System-SubAgents清单](03-Harness-System-SubAgents%E6%B8%85%E5%8D%95.md) 的 $M$ 记忆栈）→ 实现数据的**不停更新与集成**、**松散耦合**、**防排队拥塞**。

## 3. 形式化

### 3.1 数据子系统五元组

$$\mathcal{D} = \langle D_1, D_2, D_3, D_4, \,\Omega \rangle$$
其中 $\Omega$ 是 agent 自有数据 memory（$M$ 记忆栈，见 [03-Harness-System-SubAgents清单](03-Harness-System-SubAgents%E6%B8%85%E5%8D%95.md)）。

- $D_1: q \mapsto R(q)$，$q=\langle\text{src},\text{scope},\text{time},\text{proj}\rangle$（on-policy 取数）。
- $D_2: \text{src}\mapsto \Sigma(\text{src})$（off-policy 语义摘要）；检索算子 **semantic join** $\bowtie_\text{sem}: (\text{intent}, \{\Sigma\}) \mapsto \text{top-}k\ \Sigma$。
- $D_3: \text{usecase}\mapsto \pi_{\text{data}}$，数据使用策略（= data-usage skill 集合，含权重 $w$、可补充关系、需搜新源标志）。
- $D_4: (\text{data}, t)\mapsto \langle \text{validity}, \text{topic-def}_t, \text{NFR-budget}\rangle$（时新性 + 口径 + 访问预算）。

### 3.2 与 C5 三元组的接口

数据子系统经 **Harness 契约 $H$** 接入三层栈：$D_1, D_2 \subseteq \text{Interface}(H)$（Harness 直接调用），$D_3, D_4$ 为独立 API，在 context 组装阶段被 $H$ 查询以拼装"取数计划"。即扩展 §01 的契约映射：
$$H: \mathcal{I} \xrightarrow{\ \text{consult}\ D_3,D_4\ }\ \text{plan}\ \xrightarrow{\ D_1,D_2\ }\ \mathcal{E}$$

### 3.3 平面投影（承接 §05）

$$\pi(D_1)=\text{运行时DP},\quad \pi(D_2)=\text{DP(消费)}/\text{CP(生产)},\quad \pi(D_3)=\pi(D_4)=\text{运行时CP}$$

## 4. 命题扩展（P7–P9）

### 命题 P7（取数解耦 / Data-Access Decoupling）

> **在良定义 $D_3/D_4$ 控制接口下，新增数据源（$+\text{src}$）只扩大 $D_2$ 的语义覆盖与 $D_1$ 的可达切片，不改变 agent 单请求的 token 成本结构。** 形式化：
> $$\frac{\partial \ell}{\partial |\text{src}|}\Big|_{\text{固定 use case}} \approx 0$$

直觉：数据源增长是"数据面横扩"，与 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) P1 的物理扩展同构——**数据维扩展不污染逻辑维**。反例风险=把全部源 schema 灌进 context（$D_2$ 摘要泄漏进运行时数据面），缓解=semantic join 仅注入 top-$k$ 激活摘要（同 Tool Forge intent-scoped 99.2% 省 context，见 [精读-Tool-Forge-2605.28000](%E7%B2%BE%E8%AF%BB-Tool-Forge-2605.28000.md)）。

**可证伪**：固定 use case，令数据源数 $|\text{src}|: 1\to N$，测单请求 token 成本 $\ell$ → 期望基本不变；若随 $|\text{src}|$ 上升 → semantic join 未隔离原始数据，取数解耦被破坏。

### 命题 P8（语义摘要充分性 / Summary Sufficiency）

> **off-policy 语义摘要 $\Sigma$ 是 agent 正确取数的充分统计量：在 use case $u$ 上，基于 $\Sigma$ 的 semantic join + 按需 $D_1$ 拉原始数据，其结果质量不低于直接全量扫描原始数据。** 即 $\Sigma$ 对 $u$ 是 lossy-but-sufficient 的。

背景与 novelty 边界：[02-2026相关论文清单](02-2026%E7%9B%B8%E5%85%B3%E8%AE%BA%E6%96%87%E6%B8%85%E5%8D%95.md) 候选 2605.28787（*Do Agents Need Semantic Metadata?*, Halevy 等）实证"**预建语义元数据精度高 (+65.7%) 但覆盖低**、**裸 agent 在线发现覆盖高但精度低（last-mile 失败）**"。P8 主张 **off-policy 预总结**正是弥合二者的第三条路（兼顾覆盖与精度），这是该论文**明确留空**的方向。

**可证伪**：三臂对比——(a) 裸 agent 在线发现；(b) 人工 schema/元数据；(c) off-policy $\Sigma$ + semantic join。测 use case 上的 precision/recall/last-mile 成功率与 token 成本。P8 预期 (c) 在 precision 接近 (b)、coverage 接近 (a)。

### 命题 P9（治理 memory 收敛 / Governance-Memory Convergence）

> **随 use case 重复，$D_3$ 记录的 data-usage skill 使取数策略 $\pi_\text{data}$ 收敛、方差下降**：第 $n$ 次执行同类 use case 的取数决策熵 $\mathcal{H}(\pi_\text{data}^{(n)})$ 单调不增，且取数成本/质量趋稳。

直觉：$D_3$ 把"探索式取数"固化为"利用式取数"——这是 论文-Agent记忆演化综述-2605.06716 的 storage→reflection→experience 演化在**数据使用维度**的实例化（该综述只讲 agent **自身轨迹**记忆，**不讲外部数据使用治理**——P9 是其留空处的扩展）。

**可证伪**：重复运行同类 use case $n$ 次，记录每次取数路径；测决策熵与方差随 $n$ 的曲线。收敛 → 支持 P9；若发散/震荡 → $D_3$ 未能沉淀有效 data-usage skill。

## 5. 与既有研究的对照与**创新性定位**（related work + novelty）

### 5.1 既有工作覆盖了什么

| 子系统 | 最接近的工作 | 它做了什么 | 它没做什么（= 本架构 delta） |
|---|---|---|---|
| $D_1$ 取数 API | Data Product MCP (2601.08687)；Beyond Text-to-SQL DAR (2512.14622) | agent 经 MCP/in-warehouse 查询并执行访问控制 | 未与"off-policy 预总结 + 治理 memory + lifetime"统一 |
| $D_2$ 语义总结 | CoeusBI (2606.15384)；Pneuma (2601.03618 / 2604.14422)；Octopus (2601.02304)；EasyTUS (2511.02674) | 离线 agent 自动建语义视图 / 表发现 / table-union | 仍**单点**（只发现/只建视图）；无 latent-field 保留承诺、无 schema-on-read off-policy lake |
| $D_2$ 必要性论证 | **Do Agents Need Semantic Metadata? (2605.28787, Halevy 等)** | 实证元数据精度高覆盖低、在线发现覆盖高精度低 | **明确留空** off-policy 预总结这条折中路 → P8 直接填空 |
| $D_3$ 治理 memory | Five-Plane (2606.12320)；DataGovBench (2512.04416)；DPDP 框架 (2601.01101) | 治理=访问控制/合规/审计 enforcement | 治理仅"管边界"，**未把"如何用数据"沉淀为可复用 skill/memory** → $D_3$+P9 是新的 |
| $D_4$ lifetime | 记忆演化综述 (2605.06716, knowledge timeliness)；KV TTL（LLM推理-KV Cache与多租户调度） | agent **自身记忆**的时效/GC | 未覆盖**外部业务数据**口径时效 + 访问 NFR 预算 → $D_4$ 是新的组合 |

### 5.2 创新性三连（写进 paper 的 contribution）

> 用户要求"寻找创新性以增加独创性"——以下三点是 $\mathcal{D}$ 区别于所有上表工作的**独立 novelty**：

1. **N1 · off-policy 语义总结平面（schema-on-read 反 ETL 截断）**：现有 agent-data 工作要么在线发现（精度低）、要么人工元数据（覆盖低、截断 latent），**没有**"后台 off-policy agent 把源结构/内容/latent 预总结进一个不截断的 lake，再用 semantic join 按需注入"的设计。Halevy 等 (2605.28787) 的实证恰好证明这条路有空位（P8）。

2. **N2 · 数据使用即 skill：治理从"管边界"升级为"沉淀经验"**。现有数据治理（Five-Plane / DataGovBench / DPDP）只做 access-control/合规 enforcement；本架构把"agent 在某 use case 下怎样组合数据最好（top-$k$ 权重、可补充关系、何时搜新源）"记录为 **per-use-case data-usage skill**，与 task/system skill 同构（研究原则-外部记忆即Skill基础），可被 scaling agent 复用并收敛（P9）。**这是把"数据治理"和"agent 记忆/skill 理论"首次缝合**。

3. **N3 · 数据子系统沿运行时 CP/DP 自然分裂，复用同一平面不变量**：$D_1/D_2$↦数据面、$D_3/D_4$↦控制面，使 §05 的平面分离 (P4) 第三次成立——**同一套正交切分同时解释"扩展轴""控制/数据面""数据子系统"三层结构**，是架构内在一致性的强证据，也呼应"概率性控制面"（$D_2$ 的 off-policy 总结、$D_3$ 的策略学习都由 LLM 驱动 → 需可审计 + 不变量约束）。

> **一句话 novelty**：把分布式系统的 control/data plane 切分 + agent skill/memory 理论 + 数据集成（schema-on-read）三者，统一在一个 **agent-native 数据子系统**里，并给出可证伪命题 P7–P9。

## 6. 对 C5 / 主线的影响

1. **C5 维度再加一层**：逻辑/物理扩展（加什么）× 控制/数据面（怎么分流）× **数据子系统（怎么与外部数据协同）**。数据维扩展（+数据源）经 P7 与物理扩展同构。
2. **平面切分第三次复用**：$\mathcal{D}$ 沿运行时 CP/DP 分裂（N3），强化 §05 的科学性论证。
3. **skill 理论统一**：data-usage skill（$D_3$）补全 task/system skill 二分 → **三分**，均遵循同一 skill 形式。
4. **新增可证伪命题 P7–P9**：分别对应取数解耦、语义摘要充分性、治理记忆收敛。
5. **演进路径新增"数据轴"实验**（见 §7），衔接阶段 2/3。

## 7. 实验设计

| 命题 | 设计 | 自变量 | 因变量 | 通过判据 |
|---|---|---|---|---|
| **P7 取数解耦** | 固定 use case，逐步接入更多异构源 | 数据源数 $\|\text{src}\|:1\to N$ | 单请求 token 成本 $\ell$ | $\ell$ 基本不随 $\|\text{src}\|$ 上升 |
| **P8 摘要充分性** | 三臂对比（裸在线发现 / 人工元数据 / off-policy $\Sigma$+semantic join） | 取数方式 | precision / recall / last-mile 成功率 / token 成本 | (c) precision≈(b) 且 coverage≈(a) |
| **P9 治理收敛** | 重复同类 use case $n$ 次 | 重复次数 $n$ | 取数决策熵 $\mathcal{H}(\pi_\text{data})$、成本/质量方差 | 熵与方差随 $n$ 单调不增 |

数据集/平台建议：企业类用合成 ERP/CRM + 公开 table-union 基准（EasyTUS/Octopus 的数据），SaaS 类用电商时间序列，公开类用 arXiv 元数据。NFR（$D_4$）用多 agent 并发压测同一 DBMS 观察争用与空访问率。

## 8. 待验证 / 风险点

- [ ] **$D_2$ off-policy 总结自身的成本与时效**：后台总结也调 LLM，若源频繁变更，$\Sigma$ 易过期 → 与 $D_4$ lifetime 耦合，需定义"摘要再生触发阈值"（类比 $M$ 的 $\theta$）。
- [ ] **P8 的"充分统计量"如何严格定义**：不同 use case 对 $\Sigma$ 的损失敏感度不同，可能需 per-use-case 的充分性，而非全局。
- [ ] **semantic join 的检索质量**：top-$k$ 选错 → 取数错误；需评估嵌入/生成式检索在跨源摘要上的鲁棒性。
- [ ] **概率性控制面风险延伸**：$D_2$ 总结幻觉、$D_3$ 策略学偏，都会经控制面影响取数 → 需可审计日志 + 不变量（数据授权范围不可因组合越权，呼应 [精读-Benign-in-Isolation-技能组合风险-2606.15242](%E7%B2%BE%E8%AF%BB-Benign-in-Isolation-%E6%8A%80%E8%83%BD%E7%BB%84%E5%90%88%E9%A3%8E%E9%99%A9-2606.15242.md) 的 P3）。
- [ ] **$D_3$ 与 $\Omega$（自有记忆）的边界**：data-usage skill vs 一般 experience memory 如何不重复存储。
