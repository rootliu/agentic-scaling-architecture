# Agentic Runtime 参考架构 — 讨论记录与原始构想

> **来源**: 2026-06-17 与 Claude 的研究讨论（文献综述期 + 文献对照验证）
> **类型**: 研究锚点 / 原始构想存档（可被其他笔记反向链接）
> **状态**: 进行中
> **目标载体**: 学术论文 + 最终形成开源 agent 云系统

---

## 一句话定位

一个 agentic runtime 的**三层参考架构**：自底向上为 **Scaffold → Harness → Skills**，对应互联网部署的"虚机 / 软件框架运行时 / 应用前端运行时"；核心主张是 **逻辑扩展（Skills）与物理扩展（Scaffold）可以解耦**，最终目标是形成可稳定运营的、起"数字员工"作用的 agent 云系统，并作为开源软件提供给社区。

---

## 原始构想（用户原话）

> 用户原话已**逐字单独存档**于 [原始构想-用户原话](%E5%8E%9F%E5%A7%8B%E6%9E%84%E6%83%B3-%E7%94%A8%E6%88%B7%E5%8E%9F%E8%AF%9D.md)（I-1 ~ I-5），本主文档不再内嵌，以保持正文整洁、原话不可改写。
> 核心要点：以基础大模型为智力/调度基础解决具体 use case；应用需**逻辑扩展（加 specs）**与**物理扩展（加 throughput）**；自底向上三层 **Scaffold → Harness → Skills**，对应互联网部署的"虚机 / 框架运行时 / 前端运行时"；目标是可稳定运营的"数字员工"，最终形成开源 agent 云系统。

## 架构的 6 条可检验命题（claims）

| 编号 | 命题 |
|---|---|
| **C1** | agentic runtime 自底向上分为 3 层：Scaffold → Harness → Skills，且与互联网部署的"虚机 / 软件框架运行时 / 应用前端运行时"一一对应 |
| **C2** | **Scaffold 层**：最底层，对应 microVM 等隔离沙盒，提供执行环境与底层工具（bash 等） |
| **C3** | **Harness 层**：中间层，承载 memory、system prompt、context management 等"内部 agent 工具"；调用 Scaffold 工具；接入外部工具（多模态视频感知、搜索）；**并具备用大模型生成新程序工具的能力（tool synthesis）** |
| **C4** | **Skills 层**：最上层，包含 specs 描述、输出格式、工具调用条件、输出条件、循环条件；目标是**组装并迭代推送进大模型 context** |
| **C5** ⭐ | **双扩展维度**：逻辑扩展 = 增加 Skills（新 specs）；物理扩展 = 增加 Scaffold（throughput / 处理能力 / 能力细化）——两者解耦 |
| **C6** | **非功能需求驱动**：通过对各层的性能/可靠性/可用性/安全分别设计优化，使 agent 达到行业级稳定运营，承担"数字员工"角色 |

## 架构精化（2026-06-18 讨论新增，已融入主线）

> 来源原话见 [原始构想-用户原话](%E5%8E%9F%E5%A7%8B%E6%9E%84%E6%83%B3-%E7%94%A8%E6%88%B7%E5%8E%9F%E8%AF%9D.md) I-3 ~ I-5。本节把 5 段讨论提炼为对 C5 / P1 的 7 条结构性精化。

### S1 · Skills 二分：task vs system
Skills 并非同质。**task skills** 完成用户任务（留在 L3）；**system skills**（memory 压缩、整理、反思、检索）维护 runtime 自身，**归入 Harness**。判据：①输出是否进入用户可见结果；②触发源是用户意图还是运行时状态。

### S2 · Harness 内部三平面
Harness = **契约翻译（𝓘→𝓔）** + **能力供给（内/外部工具 + tool synthesis）** + **维护子系统（system skills）**。

### S3 · 维护子系统走物理轴
维护子系统对外是固定签名的函数工具，内部调 LLM 但**指令固定**；触发**∝ token 流量**（如每 32K 压缩、每 128K 写 memory），可用 Haiku 级廉价模型；随模型优化 token 输出递减 → **支持线性扩展** → 归**物理扩展**。
→ 这恰好给出 P1 "加 task skill 不拖垮 throughput" 的**机制性原因**：system skills 把"逻辑扩展带来的 context 膨胀"吸收进物理轴。

### S4 · P1 新增可证伪前提
维护操作开销**只依赖 token 量、不依赖 Skills 语义复杂度**。可实验证伪：固定 token 量、变 Skill 多样性，测压缩质量/开销是否漂移。

### S5 · 阈值参数化
压缩阈值 `θ_compact`、写回阈值 `θ_persist` 不是经验拍脑袋，而是由 **KV Cache 经济学**（显存→并发→最优 context 长度）推出（见 LLM推理-KV Cache与多租户调度）。这使阈值取值具备理论依据。

### S6 · 核心 framing：SaaS/SSR 三轴解耦
现代云应用早已"前端按流量横扩 / 业务按功能纵展 / 后端按算力扩"——**三轴解耦是已知工程范式**。本研究的 delta = **在每层内嵌概率性 LLM 调用的条件下，给出该解耦成立的充要条件（契约层 H）与可证伪判据（P1）**。维护子系统 ≈ **SSR 前端**（前端逻辑、跑在服务端、按流量扩 serving 实例）。
> 类比边界（须在论文声明）：SaaS 前端是确定性代码、零后端成本；本架构维护子系统是 LLM 驱动（非确定）、消耗推理算力。这一差异本身即 novelty。

### S7 · ClawVM 一手证据
`ClawVM`(2604.10352) 把 compaction / writeback / 校验明确定位在 **harness layer**，强调 "deterministic and auditable"。引用时须声明差异：**ClawVM 偏确定性规则，本架构偏 LLM 驱动 + 可审计日志**。详见 [03-Harness-System-SubAgents清单](03-Harness-System-SubAgents%E6%B8%85%E5%8D%95.md)。

### S8 · 第二维正交切分：控制面 / 数据面（2026-06-19 新增）
在"逻辑/物理扩展"轴之外，引入分布式系统经典的 **control plane / data plane** 正交切分。**数据面**承载 token 处理路径（∝ token 流量），**控制面**承载决策/契约翻译/维护触发/不变量（∝ 状态事件，与业务流量解耦）。两轴正交 → **2×3 设计矩阵**。收益：①借成熟系统理论削弱"工程而非科学"质疑；②给 P1 解耦提供**第二个结构性论据**（逻辑扩展作用于控制面、物理扩展作用于数据面 → 正交）；③维护子系统 $M$ 一举归位（触发在 CP、开销在 DP）。**真正 novelty = 概率性控制面**（控制决策部分由 LLM 驱动，需可审计日志 + 不变量约束其不确定性）。完整设计见 [05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md)（含命题 P4 平面分离、P5 P1 的平面分解、P6 $M$ 跨面分解）。

### S9 · 第三维：数据子系统四组成（2026-06-20 新增）
agent 与**外部数据**协同需一个**略独立于三层栈的数据子系统** $\mathcal{D}=\langle D_1,D_2,D_3,D_4,\Omega\rangle$：$D_1$ on-policy 取数 API、$D_2$ **off-policy 语义总结平面**（schema-on-read 反 ETL 截断、保留 latent、semantic join 检索）、$D_3$ **数据治理 memory**（把"如何用数据"沉淀为 per-use-case **data-usage skill**）、$D_4$ **lifetime 体系**（时新性 + 业务口径 + 访问 NFR 预算）。$D_1,D_2$ 由 Harness 直接调用（运行时数据面），$D_3,D_4$ 为独立 API、供 context 组装查询（运行时控制面）——**S8 的平面切分第三次成立**。新增可证伪命题 **P7 取数解耦 / P8 语义摘要充分性 / P9 治理记忆收敛**。**三连 novelty**：N1 off-policy schema-on-read 语义总结（填补 Halevy 等 2605.28787 留空的覆盖↔精度折中）；N2 数据治理从"管边界"升级为"沉淀 skill/memory"（**首次缝合数据治理与 agent skill/记忆理论**）；N3 数据子系统沿 CP/DP 自然分裂、复用同一平面不变量。skill 理论由 task/system **二分**扩为 task/system/data-usage **三分**。完整设计见 [06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md)。

## 把"工程"提升为"科学"的三发弹药

1. **层间契约缺失是真问题（强动机）**
   `Policy-Driven Runtime Layer`(2605.27744) 指出：上层框架懂语义、下层引擎懂事件，**跨层策略（缓存、批处理、调度）两边都需要却无人承接**。
   → 科学主张升级：**逻辑层（Skills）与物理层（Scaffold）之间需要显式契约（contract），Harness 即该契约的承载者。** 从"分层"升级为"层间接口的形式化"。

2. **组合性安全是可研究的科学问题**
   `Benign in Isolation, Harmful in Composition`(2606.15242) 证明：单 skill 安全 ≠ 组合安全。
   → C5 的"逻辑扩展（加 Skills）"必须回答：新增 Skill 时如何保持系统级不变量（invariant）？可形式化为组合性约束/定理。

3. **可扩展性可被量化**
   `Autellix`(2502.13965)、`Pythia`(2604.25899)、`AGENTSERVESIM`(2606.09613) 在量化 agent serving 的 throughput/延迟。
   → "物理扩展"可提出可扩展性度量：固定 Skills 增加 Scaffold，throughput 是否近线性？逻辑扩展对单请求延迟的边际成本？**两条曲线的正交性即 C5 的实验证据。**

> **核心一句话**: 把论文主张从"我们提出三层架构"改成
> **"我们证明 agent 的逻辑可扩展性与物理可扩展性可以解耦，解耦点是 Skills–Scaffold 之间的显式契约层（Harness）；我们给出该契约的形式化、组合性约束，并实验验证两个扩展维度的正交性。"**

## 面向学术论文 + 开源 agent 云的演进路径

**阶段 1 — 立论文（Position/Architecture）**
参考架构 + C5 解耦命题 + 与现有 serving/framework 工作的定位表；形式化层间契约；明确逻辑/物理扩展定义。

**阶段 2 — 证解耦（Empirical，核心战场）**
- 物理轴：固定 Skills，增加 Scaffold 实例 → 测 throughput 扩展曲线（对标 Autellix/Pythia）
- 逻辑轴：固定 Scaffold，增加 Skills → 测能力覆盖 vs 单请求成本
- 消融：去掉 Harness 契约层，验证两轴是否退化为耦合

**阶段 3 — 保安全/可靠（Reliability）**
组合性安全：新增 Skill 时的系统不变量保持（直接回应 2606.15242）；Scaffold 隔离保证（对标 WASM/microVM）。

**阶段 4 — 开源系统化（Systems + OSS）**
完整 agent 云系统，多租户、按层 NFR、digital-worker 运营；参照 AgentScope(2402.14034)、Agent-as-a-Service(2505.08446)、Five-Plane Governance(2606.12320)。

### C. 逻辑扩展的安全代价（→ P3 / C4）
- [精读-Benign-in-Isolation-技能组合风险-2606.15242](%E7%B2%BE%E8%AF%BB-Benign-in-Isolation-%E6%8A%80%E8%83%BD%E7%BB%84%E5%90%88%E9%A3%8E%E9%99%A9-2606.15242.md)（⭐⭐⭐）
  - **观点**：单独 benign 的 skill 组合后可能有害（Skill Composition Risk）；安全须在 **activated path 级**而非孤立工件级评估 → **P3（组合性不变量）的实证依据**。
  - **关键数据**：CapFlow 组合攻击成功率 **33.6%**（孤立近 0）；TrustLift **>96.5%**（5 backend 中 4 个）；AuthBlur 风险审批率 **+71.8%**。
  - **可用**：三风险 → 三不变量（capability containment / no trust escalation / auth isolation）；SCR-Bench 可作阶段 3 评测基线。

### D. 物理扩展 / throughput（→ C5 定义 2、P1）
- [精读-Helium-高效Agent服务-2603.16104](%E7%B2%BE%E8%AF%BB-Helium-%E9%AB%98%E6%95%88Agent%E6%9C%8D%E5%8A%A1-2603.16104.md)（⭐⭐）
  - **观点**：现有 serving 忽略 cross-call 依赖；把 workflow 当 query plan、LLM 调用当 first-class operator，可获 **1.56x** 加速 → 证明"物理扩展存在可观优化空间"，阶段 2 可把 Helium/vLLM 作 baseline。

### E. Harness 工具合成 + 解耦可行性（→ C3、P1 风险缓解、P2）
- [精读-Tool-Forge-2605.28000](%E7%B2%BE%E8%AF%BB-Tool-Forge-2605.28000.md)（⭐⭐⭐）
  - **观点**：tool = capsule（含 intent、**capability contract**、实现、测试、验证证据、生命周期、凭证、路由元数据）；Router 用 **intent-scoped session 替代全量 schema 加载，省 99.2% context**。
  - **关键作用**：①C3 tool synthesis 的落地蓝图（开源）；②"capability contract"具体化 Harness 契约接口（P2）；③**99.2% context 节省 = P1 解耦可行性的关键工程证据**（逻辑扩展不线性侵蚀 context → 不拖垮 throughput）。
  - **关键数据**：Router micro-F1 0.901；端到端 0.940；23/25 通过沙箱验证。

### F. 分层 NFR/治理的"科学化"范式（→ C6、P3）
- [精读-Five-Plane-运行时治理-2606.12320](%E7%B2%BE%E8%AF%BB-Five-Plane-%E8%BF%90%E8%A1%8C%E6%97%B6%E6%B2%BB%E7%90%86-2606.12320.md)（⭐⭐⭐）
  - **观点**：five-plane（reasoning + network/identity/endpoint/data）+ stop-anywhere mediation + composite principals/capability attenuation + audit；定义 **6 interruption primitives + 4 correctness invariants**，封堵 **7 种威胁**。
  - **关键作用**：①可作本架构 C6 安全横切层的实现；②**范式参照**——把"分层"提升为带不变量与可证明性质的科学主张（回应"工程 vs 科学"）；③capability attenuation 呼应 P3 的 capability containment。

### 跨篇综合结论（写进论文的三条）
1. **C5 仍是空白**：7 篇中无一把"扩展性"拆成逻辑/物理正交两维并绑定不同层 → novelty 内核安全。
2. **P1 解耦的工程可行性已有支撑**：Tool Forge 的 intent-scoped 加载（省 99.2% context）+ Helium 的 workflow-aware serving → 逻辑扩展不必然侵蚀物理吞吐。
3. **"科学化"路径清晰**：仿 Five-Plane（primitives + invariants + threats）与 SCR（path-aware 评估），为本架构补"不变量 + 可验证性质 + 可度量曲线"，即可破解"工程而非科学"质疑。
