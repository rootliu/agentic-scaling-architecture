## 0. 动机：为什么需要形式化

用户确认的核心 novelty 是 **C5（双扩展维度解耦）**，最担心的风险是被审稿人质疑"**工程而非科学**"。
破解之道：不停留在"我们把系统分了三层"，而是**定义两个正交的扩展维度，并证明解耦的充要条件**。
关键文献支撑：`Policy-Driven Runtime Layer`(2605.27744) 指出层间缺乏契约；本草稿即把"契约"形式化为解耦的载体。

## 1. 基本定义

设一个 agent 应用 $A$ 为三元组：

$$A = \langle S, H, X \rangle$$

- $S = \{s_1, \dots, s_m\}$：**task Skills 集合**（Specification Plane / L3，仅含面向用户任务的 skill；维护类 system skills 归入 $H$，见 §1.1）。每个 skill $s_i$ 是一个规约对象 $s_i = \langle \text{desc}_i, \sigma_i^{in}, \sigma_i^{out}, c_i^{call}, c_i^{stop}, c_i^{loop} \rangle$，分别为描述、输入 schema、输出 schema、调用条件、终止条件、循环条件。
- $H$：**Harness**（Capability/Contract Plane / L2）。承载 memory、context 管理、工具合成；本草稿将其建模为**契约映射** $H: \mathcal{I} \to \mathcal{E}$，把上层意图 $\mathcal{I}$（由 Skills 产生）翻译为下层可执行单元 $\mathcal{E}$（由 Scaffold 执行）。 内部三分：**契约翻译 + 能力供给（tool synthesis）+ 维护子系统 $M$**（system skills：压缩/写回/反思/检索），见 §1.1。
- $X = \{x_1, \dots, x_n\}$：**Scaffold 实例集合**（Execution/Isolation Plane / L1）。每个 $x_j$ 是一个隔离执行单元（microVM/sandbox），提供算力与底层工具，刻画系统的 throughput 容量。 **宽定义**：隔离 + 算力供给（含推理 serving 容量），使物理轴瓶颈可测、KV Cache 数据可作 baseline。

### 1.1 Skills 二分与维护子系统 $M$（2026-06-18 精化）

Skills 非同质，按"服务对象"二分：

| | **task skills**（留在 $S$/L3） | **system skills**（归入 $H$ 的 $M$） |
|---|---|---|
| 目的 | 完成用户任务 | 维护 runtime 自身 |
| 例 | 写代码 / 查资料 / 订票 | 压缩 / 写回 / 反思 / 检索 / GC |
| 触发源 | 用户意图 | 运行时状态（context 超长、step 数） |
| 服务 | 外部覆盖 $\mathcal{T}$ | 内部不变量 $\text{Inv}$ / 资源 |
| 扩展归属 | 逻辑轴 | **物理轴**（见 §3 P1） |

判据：①输出是否进入用户可见结果；②触发源是用户意图还是运行时状态。
$M$ 的开销特征：对外固定签名工具、内部 LLM 驱动但**指令固定**、触发 $\propto$ token 流量、可用廉价模型（Haiku 级）。故 $M$ 的成本随物理吞吐线性增长，归物理轴——这正是 P1 解耦成立的机制性原因。（平面视角：$M$ 触发逻辑在控制面、推理开销在数据面，见 §3bis P6 与 [05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md)。）

## 2. 两个扩展维度

**定义 1（逻辑扩展 Logical Scaling）.**
逻辑扩展是仅作用于 $S$ 的操作：$A = \langle S, H, X\rangle \xrightarrow{\;+s_{m+1}\;} A' = \langle S \cup \{s_{m+1}\}, H, X\rangle$。
其效果是扩大应用可解决的**问题类（task class）覆盖** $\mathcal{T}(A)$，不改变物理容量。

**定义 2（物理扩展 Physical Scaling）.**
物理扩展是仅作用于 $X$ 的操作：$A = \langle S, H, X\rangle \xrightarrow{\;+x_{n+1}\;} A'' = \langle S, H, X \cup \{x_{n+1}\}\rangle$。
其效果是提升系统**吞吐 $\Theta(A)$ / 并发处理能力**，不改变可解问题类。

**度量定义：**
- $\mathcal{T}(A)$：能力覆盖（capability coverage）——$A$ 能正确完成的任务集合（或其在某 benchmark 上的覆盖率）。
- $\Theta(A)$：吞吐（throughput）——单位时间完成的任务请求数（或 token/s），随 $|X|$ 变化。
- $\ell(A)$：单请求延迟 / 单请求 token 成本。

## 3. 核心命题

### 命题 P1（解耦 Decoupling）— 论文主命题

> **逻辑扩展（+task skill）只改变能力覆盖 $\mathcal{T}$，不改变单位 token 的维护开销；物理扩展（+Scaffold 实例 + 配套廉价维护推理 $M$）只改变吞吐 $\Theta$（总成本 = task 推理 + 维护推理，随 token 量线性增长），不改变能力覆盖。**
>
> **充要前提（2026-06-18 新增，可证伪）**：维护操作开销只依赖 token 流量、**不依赖** Skills 语义复杂度。

形式化：
$$\frac{\partial \Theta}{\partial |S|} \approx 0 \qquad\text{且}\qquad \frac{\partial \mathcal{T}}{\partial |X|} = 0$$

即两个扩展维度在效果空间 $(\mathcal{T}, \Theta)$ 上**正交**。

**实验可证伪形式**：
- 固定 $S$，令 $|X|: 1\to N$，测 $\Theta$ → 期望近线性增长（physical scaling 曲线）。
- 固定 $X$，令 $|S|: m\to m+k$，测 $\mathcal{T}$ → 期望单调增长，而 $\Theta$ 基本不变（logical scaling 曲线）。
- 若两条曲线相互独立 → 支持 P1；若加 Skills 显著拖垮 throughput → P1 被证伪（说明解耦未成立）。
- **前提证伪**：固定 token 量、变 Skill 多样性 → 测维护开销/压缩质量是否漂移；若显著漂移 → 维护开销依赖语义复杂度，物理轴线性假设被削弱。

### 命题 P2（契约充分性 Contract Sufficiency）

> **解耦（P1）成立的充分条件是：存在一个良定义的契约层 $H$，使得 Skills 仅通过 $H$ 的抽象接口访问 Scaffold，反之亦然。**

即 $S$ 与 $X$ 之间**无直接依赖**（dependency 仅经由 $H$）。
形式化为分层依赖约束：
$$\text{dep}(s_i) \subseteq \text{Interface}(H), \qquad \text{dep}(x_j) \subseteq \text{Interface}(H)$$

**消融可证伪形式**：移除/旁路 $H$（让 Skill 直接绑定特定 Scaffold 工具），观察 P1 是否退化——若退化为耦合（加 Skill 影响 throughput、加 Scaffold 需要改 Skill），则反证 $H$ 的必要性。
→ 这正是回应 `Policy-Driven Runtime Layer`(2605.27744) 的"层间无契约"问题。

### 命题 P3（组合性不变量 Compositional Invariant）

> **逻辑扩展须保持系统级安全/正确性不变量：$\forall s_{new},\ \text{Inv}(S) \Rightarrow \text{Inv}(S \cup \{s_{new}\})$（在 $H$ 强制的约束下）。**

背景：`Benign in Isolation, Harmful in Composition`(2606.15242) 证明单 skill 安全 ≠ 组合安全。
→ 因此 P3 要求 $H$ 不仅做翻译，还做**组合约束执行（invariant enforcement）**。这把 C4 的"skill 组合安全坑"转化为本架构的设计承诺。

可候选的不变量 $\text{Inv}$：资源隔离边界不被突破、工具调用权限不因组合而提升（防 privilege escalation）、输出 schema 一致性。

## 3bis. 第二维正交切分：控制面 / 数据面（2026-06-19 扩展）

> 完整设计见 [05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md)。此处仅收录回填主线的核心定义与命题。

在"扩展轴（逻辑/物理）"之外，叠加分布式系统经典的**平面切分**：数据面 DP（承载 token 处理路径，∝ token 流量）与控制面 CP（承载决策/翻译/维护触发/不变量，∝ 状态事件，与业务流量解耦）。两条切分正交，构成 **2×3 设计矩阵**：

| | **数据面 DP**（∝ token 流量） | **控制面 CP**（∝ 状态事件） |
|---|---|---|
| **L3 Skills** | 激活 skill 产生的 context tokens | skill 选择/调用/终止/循环条件判定 |
| **L2 Harness** | 工具执行流、$M$ 维护推理 token 开销 | 契约翻译 $H{:}\mathcal I\to\mathcal E$、路由、$M$ 触发策略、不变量执行 |
| **L1 Scaffold** | sandbox 计算、serving 前向 | 实例调度、放置、隔离边界设定 |

### 命题 P4（平面分离）
> 良定义契约 $H$ 下，$\dfrac{\partial L_{CP}}{\partial \dot\tau}\approx 0$ 且 $\dfrac{\partial L_{DP}}{\partial |\text{CP 规则}|}\approx 0$：业务流量不放大控制开销，控制规则复杂化不放大每请求数据面 token 成本。

### 命题 P5（P1 的平面分解）
> 逻辑扩展的一阶效应落在控制面（扩大 CP 决策空间），物理扩展的一阶效应落在数据面（扩大 DP 吞吐容量）；作用平面不同 → 效果正交。这是 P1 的**第二个结构性来源**（独立于 P2 的契约层论据）。P1 的反例"加 Skills 撑大 context 拖垮 throughput"在此被定位为**控制面逻辑泄漏进数据面**，缓解=skill 规约留 CP、仅按需把激活路径注入 DP（即 Tool Forge intent-scoped 加载的平面学解释）。

### 命题 P6（维护子系统跨面分解）
> $M$ 的触发逻辑属控制面（$\theta$ 阈值 + 状态事件，频率与业务流量解耦），推理开销属数据面（随物理吞吐线性）。故 $M$ 是平面分离的典型实例，而非反例。

> **novelty 声明**：经典系统控制面是确定性配置下发；本架构控制面部分由 LLM 驱动（契约翻译/反思/tool synthesis），是**概率性控制面（probabilistic control plane）**——需用可审计日志 + 不变量执行约束其不确定性。这一差异是平面切分的真正贡献点。

## 3ter. 第三维：数据子系统（2026-06-20 扩展）

> 完整设计见 [06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md)。此处仅收录回填主线的核心定义与命题。

在"扩展轴""控制/数据面"之外，agent 与**外部数据**协同需要一个**略独立于三层栈的数据子系统** $\mathcal{D} = \langle D_1, D_2, D_3, D_4, \Omega\rangle$：$D_1$ on-policy 取数 API、$D_2$ off-policy 语义总结平面（schema-on-read，反 ETL 截断，semantic join 检索）、$D_3$ 数据治理 memory（把"如何用数据"沉淀为 per-use-case **data-usage skill**）、$D_4$ lifetime 体系（时新性 + 业务口径 + 访问 NFR 预算）。其中 $D_1,D_2$ 由 Harness 直接调用（运行时数据面），$D_3,D_4$ 为独立 API、供 context 组装时查询（运行时控制面）——**§3bis 的平面切分在此第三次成立**。

契约映射扩展为：$H: \mathcal{I} \xrightarrow{\text{consult } D_3,D_4} \text{plan} \xrightarrow{D_1,D_2} \mathcal{E}$。

- **P7（取数解耦）**：固定 use case，$\partial\ell/\partial|\text{src}|\approx 0$——加数据源不污染单请求 token 成本（与 P1 物理扩展同构）。
- **P8（语义摘要充分性）**：off-policy 摘要 $\Sigma$ 是取数的充分统计量，兼顾覆盖与精度——填补 *Do Agents Need Semantic Metadata?*(2605.28787) 留空的折中路。
- **P9（治理记忆收敛）**：随 use case 重复，data-usage skill 使取数决策熵单调不增。
- **三连 novelty**：N1 off-policy schema-on-read 语义总结；N2 数据治理从"管边界"升级为"沉淀 skill/memory"（缝合数据治理与 agent skill 理论）；N3 数据子系统沿 CP/DP 自然分裂、复用同一平面不变量。

> skill 理论由此从 task/system **二分**扩为 task/system/data-usage **三分**，三者同遵 skill 形式 $\langle\text{desc},\sigma^{in},\sigma^{out},c\rangle$。

## 3quater. 第四维：并行度与局部性协同（2026-07-10 扩展）

> 完整设计见 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md)。此处仅收录回填主线的核心命题。

在三维解耦（不劣化）之上，给定任务总预算 $B$，还需**主动最大化 budget 内的并行吞吐**。手段是一套跨三层的"高内聚局部性 + 跨单元松散耦合"协同设计：**M1** 按工具集划分逻辑子 agent（Skills 局部性→attention 命中 $H\uparrow$）；**M2** 子 agent 间工具集**不重合**切分（并行独立性→串行比 $s\downarrow$）；**M3** 子 agent 间只传 **summarized context**、detail 入 memory（编排效率 $E\uparrow$，松散耦合顺序关系）；**M4** Scaffold 按**派生种子 seed** 亲和派生（同种共置复用、异种分散）。一根杠杆——**工具集切分**——同时定义 Skills 局部性、并行独立性与 Scaffold seed 边界。

- **P10（并行度上界可设计）**：$s=s_0+\lambda\rho$、$E=E_0-\mu\bar c_{ctx}$，故吞吐随工具集重合度 $\rho\downarrow$、跨 agent 传递量 $\bar c_{ctx}\downarrow$ 逼近阿姆达尔上界 $1/s$——并行度上界是切分设计的函数，而非外生常数（把 [08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) 启发式 III 从被动上界升级为可设计上界）。
- **P11（工具集局部性）**：子 agent 工具集越稳定，context 跨迭代前缀命中率 $H$ 越高、单位有效 token 成本越低。
- **P12（seed 亲和局部性）**：同 seed 共置 + 异 seed 分散，在固定 Scaffold 资源（CPU/安全服务容量）下可容纳的并行子 agent 数 $k_{\max}$ 高于随机放置。
- **P13（规划期局部性预探测）**：主 agent 规划时 dry-run 每个候选子任务的工具集闭包 $\hat\tau$ 与 Scaffold 环境范围，据此切分，使划分的工具集重合度 $\rho$ 更低、前缀命中 $H$ 更高（类比编译器局部性感知调度 / cost-based query planning）。

> **机制根因（§0.5 of [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md)）**：推理=内在知识⊕context，agent 唯一能动的是 context；context 变更是 KV-cache 失效根因，故"前缀逐 token 冻结 + 尾部变化"同时买到缓存复用（快）、注意力聚焦（准）、并行独立（可切分）。P10–P13 是此根因的四个可证伪投影。

> 因数据子系统 $\mathcal{D}$ 栈外、detail 走 memory，三层栈内的并行耦合始终松散——这使 P10–P12 复用同一"内聚高频 / 松散低频"不变量（架构一致性第四次成立）。

## 3quinque. 第五维（正交表示形态）：Skill-as-Code 与跨模型稳定（2026-07-12 扩展）

> 完整设计见 [10-Skill-as-Code与确定性固化](10-Skill-as-Code%E4%B8%8E%E7%A1%AE%E5%AE%9A%E6%80%A7%E5%9B%BA%E5%8C%96.md)。此处收录回填主线的核心命题。

前四维是「怎么解耦 / 怎么并行」；本节是**正交的表示形态维**——同一 skill 可为 LLM 形态或 **code 形态**。当 skill 操作空间闭合（工具集 $\tau$ 闭合、$\sigma^{out}$ 固定、分支可枚举），其可固化度 $\kappa(s)$ 超阈，Harness 把其「获取→判断→核验→决策」过程固化为 **code**，使这段执行**不再调用模型权重 $\theta$**。code 是概率控制面里的**确定性来源**（N8），也是模型版本演进下的**契约稳定锚**。

- **P14（跨模型稳定性）**：$\partial\,\text{output}/\partial\theta\big|_{s^{\text{code}}}\approx 0$ 且 $\text{Var}_\theta(s^{\text{code}})\ll\text{Var}_\theta(s^{\text{LLM}})$——被 code 固化的 skill 输出与模型版本 $\theta_t\to\theta_{t+1}$ 解耦。**这是 P2（契约充分性）在时间维的扩展**：P2 让 $S$ 与 $X$ 在空间上解耦，P14 让 skill 输出与 $\theta$ 在版本上解耦。可证伪：跨 $\geq3$ 模型版本重跑同一 use case，code skill 方差不低于纯 LLM skill 则被证伪。
- **确定性梯度设计律**：确定性沿 agent loop 阶段是梯度——**派生/执行端向 code 收敛**（要可复现、抗漂移），**reflection/校验端向 NL 发散**（要泛化，NL 兼作多样性校验与随机反例 shots）。见 [10-Skill-as-Code与确定性固化](10-Skill-as-Code%E4%B8%8E%E7%A1%AE%E5%AE%9A%E6%80%A7%E5%9B%BA%E5%8C%96.md) §4。

> 机制根因（承 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) §0.5）：agent 唯一能动的是 context；code 化把「决策过程」整段冻结，与 P11「前缀逐 token 冻结」对偶——前者省整段概率生成、抗 $\theta$ 漂移，后者省 KV 重算。二者是同一「冻结稳定部分」杠杆的不同投影。

## 3quinque. 第五维（正交表示形态）：Skill-as-Code 与跨模型稳定（2026-07-12 扩展）

> 完整设计见 [10-Skill-as-Code与确定性固化](10-Skill-as-Code%E4%B8%8E%E7%A1%AE%E5%AE%9A%E6%80%A7%E5%9B%BA%E5%8C%96.md)。此处收录回填主线的核心命题。

前四维是「怎么解耦 / 怎么并行」；本节是**正交的表示形态维**——同一 skill 可为 LLM 形态或 **code 形态**。当 skill 操作空间闭合（工具集 $\tau$ 闭合、$\sigma^{out}$ 固定、分支可枚举），其可固化度 $\kappa(s)$ 超阈，Harness 把其「获取→判断→核验→决策」过程固化为 **code**，使这段执行**不再调用模型权重 $\theta$**。code 是概率控制面里的**确定性来源**（N8），也是模型版本演进下的**契约稳定锚**。

- **P14（跨模型稳定性）**：$\partial\,\text{output}/\partial\theta\big|_{s^{\text{code}}}\approx 0$ 且 $\text{Var}_\theta(s^{\text{code}})\ll\text{Var}_\theta(s^{\text{LLM}})$——被 code 固化的 skill 输出与模型版本 $\theta_t\to\theta_{t+1}$ 解耦。**这是 P2（契约充分性）在时间维的扩展**：P2 让 $S$ 与 $X$ 在空间上解耦，P14 让 skill 输出与 $\theta$ 在版本上解耦。可证伪：跨 $\geq3$ 模型版本重跑同一 use case，code skill 方差不低于纯 LLM skill 则被证伪。
- **确定性梯度设计律**：确定性沿 agent loop 阶段是梯度——**派生/执行端向 code 收敛**（要可复现、抗漂移），**reflection/校验端向 NL 发散**（要泛化，NL 兼作多样性校验与随机反例 shots）。见 [10-Skill-as-Code与确定性固化](10-Skill-as-Code%E4%B8%8E%E7%A1%AE%E5%AE%9A%E6%80%A7%E5%9B%BA%E5%8C%96.md) §4。

> 机制根因（承 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) §0.5）：agent 唯一能动的是 context；code 化把「决策过程」整段冻结，与 P11「前缀逐 token 冻结」对偶——前者省整段概率生成、抗 $\theta$ 漂移，后者省 KV 重算。二者是同一「冻结稳定部分」杠杆的不同投影。

## 4. 与互联网部署的类比（C1，作为直觉而非证明）

| 本架构层 | 互联网部署对应 | 扩展语义 |
|---|---|---|
| Skills (L3, Specification) | 应用前端运行时 | 加功能/页面 = 逻辑扩展 |
| Harness (L2, Capability/Contract) | 软件框架运行时 | 框架契约稳定 → 前后端解耦 |

## 5. 待验证 / 风险点

- [ ] **P1 的 $\partial\Theta/\partial|S|\approx 0$ 在 context 增长时是否成立？** 加 Skills 会增大 system prompt/context，可能间接抬高单请求成本 $\ell$，进而影响 $\Theta$。需区分"Skills 数量"与"激活的 Skills context 长度"——可能需要引入"按需加载 Skills"假设（与 context 管理/KV cache 相关，见 LLM推理-KV Cache与多租户调度）。
- [ ] **能力覆盖 $\mathcal{T}$ 如何客观度量？** 需选定 benchmark；逻辑扩展曲线的 x 轴是 skill 数还是 task class 数。
- [ ] **P2 的"无直接依赖"在 tool synthesis（Harness 用 LLM 生成新工具）场景下是否仍成立？** 生成的新工具运行在 Scaffold，但由 Harness 产生——需厘清这是否破坏分层。
- [ ] P3 的不变量集合需要可形式化、可检验，否则又退回"工程承诺"。
