## 0. 目标函数：budget 内吞吐最大化

给定一次任务的总预算 $B$（token / 时间 / 资源上限），系统应最大化在 $B$ 内**并行完成的有效工作量**。沿用 [08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) §4 启发式 III（代理加速受**串行比 $s$** 与**编排效率 $E$** 约束，阿姆达尔上界）：

$$\text{Speedup}(k) = \frac{1}{s + \frac{1-s}{k}\cdot E^{-1}}\quad\xrightarrow{k\to\infty}\quad \frac{1}{s}$$

- $k$ = 并行子 agent 数；$s$ = 不可并行的串行比例；$E$ = 编排效率系数（含通信/上下文传递开销）。
- **关键洞察**：盲目加 $k$ 收益受 $1/s$ 封顶，且 $E<1$ 会因通信开销进一步吃掉收益。**因此提升吞吐的真正杠杆不是加 $k$，而是同时压低 $s$ 和抬高 $E$。**

本文主张：**$s$ 与 $E$ 都可以通过"按工具集切分子 agent + summarized-context 传递 + seed 亲和派生"这套协同设计来主动改善**——这就是把阿姆达尔上界"设计出来"，而非被动接受。

> 与前三维的关系：P1/P4/P7 保证"扩展维度解耦"（加东西不互相拖累）；本文 P10–P12 进一步保证"**在解耦之上，把并行度推到 budget 允许的上界**"。前者是"不劣化"，后者是"主动最优化"。

## 0.5 机制根因：推理 = 内在知识 ⊕ context，agent 的本职是"组织 context"

> 本节是 P10–P13 的**微观第一性原理**，把并行/局部性从"工程启发式"下沉到"推理机制根因"。来源：2026-07-11 讨论。

**第一性原理**：一次 LLM 推理的输出，由两部分组合决定——**模型内在知识**（权重，固定）⊕ **context**（可变输入）。模型权重在一次任务内不变，故 **agent 能操纵的唯一变量就是 context**。因此：

> **agent 的本职工作 = 组织 context。** 规划、取数、调用工具、写 memory，本质都是在决定"下一次推理时 context 里放什么、放的顺序"。

**这带来一个被忽视的代价结构**。agent 循环每一轮都在改写 context，而 context 的改写正是 **KV-cache 失效的根因**：

- KV-cache（prefix caching：vLLM / SGLang RadixAttention）复用的是**已算过 token 的 K/V 张量**；复用前提是**共享前缀逐 token 完全相同**。
- 由于 causal attention + 位置编码，**某位置 token 一变，其后所有位置的 K/V 全部失效并需重算**。
- 推论（比"context 变化要小"更强的约束）：**能被复用的不是"变化小的 context"，而是"变化只发生在尾部、前缀被逐 token 冻结的 context"**。

**由此重新理解长程任务失控**：一个 agent 若在异构工具/目标间频繁跳变，则每轮 context 前缀被大幅重写 → (1) KV 前缀命中率 $H$ 崩、单位有效 token 成本升；(2) context 越长越杂，相关信息的注意力被稀释、中段易被忽略（lost-in-the-middle / context rot）→ **准确率与可控性同时下降**。这就是"长程任务可控性下降"的机制解释。

> **设计律（context 布局）**：把**稳定不变的部分（system prompt / 工具集 schema / 固定 memory）放在 context 最前端并逐 token 冻结**，把**每轮变化的任务内容放尾部**。前缀冻结区越长、越稳定，$H$ 越高。这直接支撑 M1（工具集稳定）并把 [08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) 启发式 I（语义局部性 $S=1/((1-H)+H\alpha^{-1})$）从"经验命中率"落成"可实现的 context 布局规则"。

**两条正交收益（务必分开，勿混为"注意力更快"）**：

1. **速度收益 = context 更短（$N\downarrow$）**：聚焦工具集 → context 短 → attention 的 $O(N^2)$ 计算与 KV 显存 $O(N)$ 下降 → 单请求更快，且固定显存下可容纳更多并行请求（$k_{\max}\uparrow$，接 P12）。**速度来自序列长度 $N$，不来自"注意力更聚焦"。**
2. **准确收益 = 注意力不被稀释**：无关内容少 → 相关 token 注意力权重不被摊薄、中段不易被忽略 → 准确率升。

> **一句话**：三维解耦让"加东西不劣化"，而本维的根因在于——**既然 agent 唯一能动的是 context，那么把 context 组织成"高内聚前缀冻结 + 松耦合尾部变化"，就同时买到了缓存复用（快）、注意力聚焦（准）与并行独立（可切分）三重收益。** P10–P13 是这一根因在四个可证伪面上的投影。

## 1. 三层各自的局部性与并行约束

一次任务被拆成多个子 agent 并行，但并行度在每层都受限，且每层都有一个可优化的"局部性"目标：

| 层 | 并行约束（上界来源） | 局部性目标（要最大化的重用） |
|---|---|---|
| **L3 Skills** | context 组装质量：子 agent 每轮迭代的 context 若与上一版差异大，则 **attention/KV 命中率崩** | **工具集局部性**：一个子 agent 尽量**反复使用同一套工具集/harness**，使 context 跨迭代高度相似（启发式 I 的命中率 $H\uparrow$） |
| **L1 Scaffold** | 物理资源上限：CPU 数、安全管理服务容量、serving 并发 | **seed 资源局部性**：**同种**子 agent 复用同一**派生种子 seed** 的资源（同虚机/镜像/缓存热区），**不同种**子 agent 资源**分散** |
| **栈外 数据子系统 𝒟** | 不在三层内 → **不构成并行耦合**（数据基盘天然松散） | detail 沉淀在 memory（$D_1/D_2/\Omega$），子 agent 间只传 **summarized context** |

> **总设计律**：**同种子 agent 内聚**（各自最大化 attention/资源局部性）+ **跨子 agent 松散耦合**（工具集不重合、只传摘要、资源分散）。这同时压低串行比 $s$（不重合 → 少同步）与抬高编排效率 $E$（传摘要而非全量 → 少通信），逼近 $1/s$ 上界。

## 2. 设计系列（三层协同的四个机制）

### 机制 M1 · 按工具集划分逻辑子 agent（Skills 层，服务局部性）

- **切分准则**：以**工具集 $\tau(a)$** 为子 agent $a$ 的边界——一个子 agent 在其生命周期内尽量只调用**同一套工具/harness 契约**。
- **效果**：子 agent 每轮迭代注入 context 的工具 schema / 调用范式高度稳定 → context 跨迭代**前缀高度可重用** → KV 前缀缓存命中率 $H\uparrow$（启发式 I）。
- **反模式**：一个子 agent 在异构工具间频繁跳变 → 每轮 context 大幅重写 → 缓存失效、$H\downarrow$。

### 机制 M2 · 子 agent 间按工具集**不重合**切分（提升并行度）

- **切分准则**：子 agent 集合 $\{a_1,\dots,a_k\}$ 的工具集**尽量两两不相交**：$\tau(a_i)\cap\tau(a_j)\approx\varnothing$。
- **效果**：工具集不重合 ⇒ 资源/权限争用少 ⇒ 可并行的"独立性"高 ⇒ **串行比 $s\downarrow$**。工具集是"逻辑资源"的代理变量：不重合的逻辑资源天然可并行。

### 机制 M3 · summarized-context 传递 + detail 入 memory（松散耦合的顺序关系，抬高 $E$）

- **传递准则**：子 agent 之间**只传 summarized context**（结论/接口/关键约束），**detail 放进 memory**（数据子系统 $D_1/D_2/\Omega$，见 [06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md)），后续子 agent **按需** semantic-join 回取。
- **效果**：跨 agent 通信量从 $O(\text{全量 context})$ 降到 $O(\text{摘要})$ ⇒ **编排效率 $E\uparrow$**；形成**松散耦合的顺序 agent 关系**（loosely-coupled sequential agents）——顺序依赖只通过薄摘要接口传递，detail 依赖被 memory 吸收，故仍可在依赖允许处并行。
- **与 𝒟 的接口**：detail 落 $\Omega$（agent 自有 memory）或经 $D_1/D_2$；因数据基盘**栈外、不耦合进三层**（[06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md) §0），这种"摘要传递 + detail 外置"天然松散。

### 机制 M4 · seed 亲和派生（Scaffold 层，物理局部性 + 分散）

- **派生准则**：每**种**子 agent 有自己的**派生种子** $\text{seed}(a)$（镜像 + 依赖 + 工具集预热 + 缓存热区的模板）。
  - **同种亲和（affinity）**：同 $\text{seed}$ 的子 agent 资源使用尽量**接近并复用**——放同一台虚机/同一资源域，复用镜像层、工具进程、KV 缓存热区。
  - **异种反亲和（anti-affinity）**：不同 $\text{seed}$ 的子 agent 资源尽量**分散**到不同虚机/资源域，降低争用。
- **效果**：同种内聚 ⇒ 物理局部性高（冷启动少、缓存命中高）；异种分散 ⇒ Scaffold 层松散耦合、并行不互扰 ⇒ 在固定 CPU/安全服务容量下容纳更多并行子 agent。

> **四机制合力**：M1（Skills 内聚）+ M4（Scaffold 内聚）把**同种子 agent 的 attention/资源局部性拉满**；M2（工具集不重合）+ M3（摘要传递）把**跨子 agent 耦合降到最松**。前者最大化各自命中率、后者最大化整体并行——**两个目标经"工具集切分"这一根杠杆同时达成**（工具集既定义 Skills 局部性边界、又定义 Scaffold seed 边界、还定义并行独立性边界）。

## 3. 形式化与命题（P10–P13）

沿用 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) 的 $A=\langle S,H,X\rangle$。引入：

- **子 agent** $a$：一个带工具集 $\tau(a)\subseteq$ Harness 能力、运行在某 Scaffold 实例 $x\in X$ 上的执行单元。
- **派生种子** $\text{seed}(a)$：$a$ 的资源模板（镜像/依赖/工具预热/缓存热区）。同种 $\Leftrightarrow$ 同 seed。
- **子 agent 划分** $\mathcal{P}=\{a_1,\dots,a_k\}$：对一个任务的并行分解。
- **工具集重合度** $\rho(\mathcal{P}) = \dfrac{\sum_{i<j}|\tau(a_i)\cap\tau(a_j)|}{\sum_{i<j}|\tau(a_i)\cup\tau(a_j)|}$（Jaccard 平均，$\in[0,1]$）。
- **context 跨迭代相似度** $H(a)$：$a$ 相邻迭代 context 的前缀可重用率（≈ KV 前缀命中率）。

### 命题 P10（并行度上界可设计 / Designable Parallelism Bound）— 本文主命题

> **通过按工具集不重合切分（$\rho\downarrow$）与 summarized-context 传递（$E\uparrow$），可把代理加速的串行比 $s$ 主动压低、编排效率 $E$ 主动抬高，使 budget $B$ 内的有效吞吐逼近阿姆达尔上界 $1/s$——即并行度上界不是外生常数，而是切分设计的函数。** 形式化：
> $$s = s_0 + \lambda\cdot\rho(\mathcal{P}),\qquad E = E_0 - \mu\cdot \bar{c}_{\text{ctx}}(\mathcal{P})$$
> 其中 $\bar{c}_{\text{ctx}}$ 为跨 agent 平均传递上下文量。故 $\text{Speedup}$ 随 $\rho\downarrow$（工具集越不重合）、$\bar c_{\text{ctx}}\downarrow$（越只传摘要）而单调增。

**可证伪**：固定任务与 $B$，构造两组划分——(a) 工具集高重合 + 传全量 context；(b) 工具集不重合 + 只传摘要。测端到端 wall-clock / budget 内完成子任务数。P10 预期 (b) 显著优于 (a)；若无差异 → 切分设计对并行度无因果影响，P10 被证伪。

### 命题 P11（工具集局部性 → 冻结前缀命中 / Tool-Set Locality）

> **重定义 $H(a)$**（承 §0.5）：$H(a) = \dfrac{\text{逐 token 冻结的前缀长度}}{\text{总 context 长度}}$（≈ KV prefix cache 命中率），**而非模糊的"相似度"**——因为 KV 复用要求前缀**逐 token 相同**，尾部一变则其后全失效。
>
> **子 agent 在其生命周期内工具集越稳定（M1）、且把稳定部分布局在 context 前缀并逐 token 冻结，其 $H(a)$ 越高，单位有效 token 成本越低。** 即 $H(a)$ 随 $a$ 的工具集切换频率、以及前缀改写频率单调下降。

**可证伪**：对同一子任务，比较三组——(i) 固定工具集 + 稳定前缀布局；(ii) 固定工具集但前缀频繁改写（把变化内容插到前端）；(iii) 跨异构工具跳变。P11 预期 (i) 的 KV 前缀命中率显著高于 (ii)、(ii) 高于 (iii)；若 (i)≈(iii) → 工具集稳定性 / 前缀冻结与缓存局部性无关，P11 被证伪。（这是 [08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) 启发式 I 的可操作化。）
### 命题 P12（seed 亲和资源局部性 / Seed-Affinity Locality）

> **同 seed 子 agent 共置（affinity）+ 异 seed 分散（anti-affinity），在固定 Scaffold 资源（CPU/安全服务容量）下，可容纳的并行子 agent 数 $k_{\max}$ 高于随机放置。** 即物理局部性把"资源上限"翻译成"更高的可并行度"。

**可证伪**：固定资源池，比较 seed 亲和放置 vs 随机放置的 $k_{\max}$ 与冷启动/缓存命中开销。P12 预期亲和放置 $k_{\max}$ 更大、冷启动更少；若无差异 → seed 亲和无物理局部性收益，P12 被证伪。

> **与前三维的关系（架构一致性第四次复用）**：P10–P12 仍复用同一"局部性/松散耦合"不变量——**高频内聚路径**（同种子 agent 的 context/资源）与**低频跨界路径**（跨 agent 摘要传递）分离。这与 [07-创新点总结](07-%E5%88%9B%E6%96%B0%E7%82%B9%E6%80%BB%E7%BB%93.md) §2 的"高频在线 / 低频离线"分离**同构**，是**第四条正交切分**（并行/局部性切分）。

## 4. Walkthrough：一次任务的并行编排（端到端）

> 场景：用户任务"分析某电商 3 个 ASIN 近三周库存与销量关联，产出可视化报告"。展示四机制如何协同。

```
[T0] 意图进入 Harness H
     H: consult D3(数据使用策略), D4(口径/NFR预算)  ← 运行时控制面(CP)
     H 生成"取数计划 + 子agent划分方案"
     ├─ 按工具集不重合切分(M2): ρ→低
     │    a1 = {数据取数工具集}    seed_data
     │    a2 = {统计/分析工具集}   seed_analysis
     │    a3 = {可视化/报告工具集}  seed_report
     └─ 派生方案(M4): 同种亲和、异种反亲和

[T1] Scaffold X 按 seed 派生(M4)
     a1 → VM#1 (seed_data, 预热取数工具/缓存热区)
     a2 → VM#2 (seed_analysis, 分散)
     a3 → VM#3 (seed_report, 分散)
     * 若并行需要 2 个 a1 实例 → 同 VM#1 亲和复用镜像/工具进程(affinity)

[T2] a1 并行取数 (运行时DP)
     a1 反复用同一取数工具集(M1) → context 跨迭代高度相似 → KV命中H↑(P11)
     经 D1 on-policy 取原始数据；经 D2 semantic-join 取语义摘要
     detail(原始表/明细) → 落 memory Ω/lake(M3)
     a1 只向下游传 summarized-context: "3 ASIN×21天 库存&销量 已就绪, 句柄=…"

[T3] a2 接 a1 的摘要(M3, 非全量) → 统计分析
     需要明细时: semantic-join 从 memory 按需回取(松散耦合顺序关系)
     a2 用固定分析工具集(M1) → H↑
     a2 传摘要给 a3: "相关系数/异常点/结论要点"

[T4] a3 接摘要 → 生成可视化报告
     a3 固定报告工具集(M1)

[编排层持续] H 在 CP 侧:
     - 依赖允许处并行(a1 多切片并行), 依赖处顺序(a1→a2→a3)
     - 不变量执行 Inv(权限/预算 B 不超)  ← 确定护栏门(见 08 §2)
     - budget 计量: token/资源累计 ≤ B

[结果] 因 ρ低(M2)+摘要传递(M3) → s↓E↑ → budget B 内完成, 逼近 1/s (P10)
       因 seed 亲和(M4) → 固定 CPU/安全容量下容纳更多并行(P12)
```

**关键点**：数据 detail 全程走**栈外数据子系统 𝒟**（memory/lake），子 agent 间只流动薄摘要——所以三层栈内的并行耦合始终是**松散**的（数据基盘不参与三层耦合）。

## 5. Specs（可实现接口草案）

> 供工程实现与实验对齐。所有 spec 遵循 skill 形式 $\langle\text{desc},\sigma^{in},\sigma^{out},c\rangle$，与 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) §1 一致。

### Spec S-1 · 子 agent 划分器（Harness/CP 侧）
```
partition(task, D3, D4, budget B, dryrun=probe(τ̂, env_scope)) -> Plan   # P13: 规划期局部性预探测
  Plan = {
    subagents: [ {id, tool_set τ, seed_ref, role} ],
    edges: [ {from, to, contract: SummarizedContextSchema} ],  # 顺序/依赖 DAG
    parallel_groups: [id...](id....md),   # 可并行组(依赖 DAG 的反链)
  }
  目标: minimize ρ(Plan)  s.t.  covers(task) ∧ Σcost ≤ B     # M2, P10
  约束: 每个 subagent 的 τ 尽量闭包(生命周期内不跳变)          # M1, P11
        稳定部分布局在 context 前缀并逐token冻结(H↑)          # §0.5, P11
  probe(): 轻量 dry-run 估 τ̂(工具集闭包)+env_scope(Scaffold范围), 无副作用   # P13
```

### Spec S-2 · 派生种子（Scaffold/CP→DP）
```
seed := ⟨image, deps, tool_prewarm(τ), cache_hint⟩
derive(seed, affinity_key) -> instance x ∈ X
  placement:
    同 affinity_key(同种)  -> co-locate(affinity)   复用镜像/工具进程/KV热区   # M4 affinity, P12
    异 affinity_key(异种)  -> spread(anti-affinity)  分散降争用                # M4 anti-affinity
  资源守恒: Σ active instances ≤ Scaffold capacity(CPU, 安全服务容量)
```

### Spec S-3 · summarized-context 传递契约（跨子 agent）
```
SummarizedContextSchema = ⟨
  claims,            # 结论/关键结果(进下游 context)
  handles,           # 指向 memory 的 detail 句柄(D1 query / D2 Σ / Ω ref)
  constraints,       # 传递的约束/不变量(供 Inv 检查)
⟩
emit(subagent, summary): write detail→memory(Ω/lake), pass summary only   # M3, E↑
consume(subagent, summary): if need detail → semantic_join(handles)       # 按需回取
```

### Spec S-4 · 局部性度量(实验用)
```
H(a)   = KV_prefix_hit_rate(a)                    # P11 指标
ρ(P)   = mean Jaccard(τ(a_i), τ(a_j))             # P10 指标(越低越好)
c̄_ctx  = mean cross-agent transferred tokens       # P10 指标(越低越好)
k_max  = max concurrent subagents under capacity   # P12 指标
```

## 6. 对主线 / novelty 的影响

1. **第四条正交切分**：并行/局部性切分（"如何在 budget 内并行"），与扩展轴、平面、数据子系统正交，且**复用同一解耦不变量**（内聚高频 / 松散低频）——[07-创新点总结](07-%E5%88%9B%E6%96%B0%E7%82%B9%E6%80%BB%E7%BB%93.md) §2 的架构一致性论证获得第四个实例。
2. **主命题 P10** 把 ICA 启发式 III 从"被动上界"升级为"**可设计上界**"，是本轮讨论最强的新科学主张。
3. **一根杠杆统三层**：工具集切分同时定义 Skills 局部性(M1)、并行独立性(M2)、Scaffold seed 边界(M4)——**架构简洁性**的强证据。
4. **N6（拟）**：并行度可设计 + 局部性协同（seed 亲和 + 工具集切分 + 摘要传递），对标 ICA 启发式 III 未给出的"如何主动逼近上界"。
5. **N7（拟）**：**规划期局部性预探测（dry-run 切分）**——把切分决策的输入从"任务语义"扩展到"工具集闭包 + Scaffold 环境可行性"，类比编译器局部性感知调度 / cost-based query planning。现有多 agent 编排按语义/角色切分，未把"规划期资源局部性 dry-run"识别为切分依据（P13）。
6. **机制根因下沉**（§0.5）：把 P10–P13 统一归因到"推理=内在知识⊕context、agent 本职是组织 context、context 变更=KV 失效根因"，使命题从工程启发式升级为有推理机制根因的科学主张。

## 7. 待验证 / 风险点

- [ ] **工具集不重合与任务可分性冲突**：有些任务本质工具集重合（都要读同一 DB），$\rho$ 难降。需界定"可切分性"前提，或用 D3 的 data-usage skill 缓存共享取数结果。
- [ ] **摘要充分性**：M3 只传摘要，若摘要丢关键 detail 导致下游反复回取 → $E$ 反降。依赖 P8（[06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md) 摘要充分性）成立。
- [ ] **seed 亲和 vs 反亲和的调度冲突**：亲和(省缓存)与分散(降争用)有张力，需一个放置目标函数权衡（借鉴 K8s affinity/anti-affinity）。
- [ ] **概率性划分的稳定性**：划分器本身是 LLM(概率性控制面 N0)，同一任务两次划分可能不同 → 需 D3 沉淀"好的划分方案"为 data-usage/orchestration skill(P9 收敛)。
  - **确定性梯度（部分正解，见 [10-Skill-as-Code与确定性固化](10-Skill-as-Code%E4%B8%8E%E7%A1%AE%E5%AE%9A%E6%80%A7%E5%9B%BA%E5%8C%96.md)）**：当划分/派生达到操作空间闭合时，把这段固化为 code（N8/P14）→ 派生/执行端确定化、抟 θ 漂移；而全局 reflection/校验端保留 NL 概率生成（多样性校验 + 随机反例）保泛化。
- [ ] **dry-run 自身成本**：P13 的规划期 dry-run 若过重，其探测成本会吃掉切分收益；需限定为"轻量、无副作用、可缓存"的探测（工具 schema 静态分析 + 历史 use case 命中）。
- [ ] **$s_0$ 的下界**：即使 $\rho\to0$，任务固有串行(如 a1→a2→a3 报告链)仍设定 $s_0>0$；P10 只逼近 $1/s_0$ 而非无限。
