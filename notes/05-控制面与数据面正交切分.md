# 05 — 控制面 / 数据面：第二维正交切分

> **来源**: 2026-06-19 研究讨论，从 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) 的 P1 机制解释展开
> **类型**: 论文核心理论扩展（formalization extension）
> **状态**: 草稿 v0.1
> **定位**: 引入分布式系统经典的 **control plane / data plane** 正交切分，与"逻辑/物理扩展"轴交叉，构成 agentic runtime 的**二维设计框架**；为 P1 解耦提供第二层机制性解释，并把"工程分层"进一步抬升为"可借用成熟系统理论的科学主张"。

---

## 0. 动机：为什么再引入一维

[01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) 已经沿"扩展方式"把系统切成**逻辑轴（Skills/𝒯）**与**物理轴（Scaffold/Θ）**。但这一维回答的是"**往哪个方向加东西**"，没有回答"**系统内部按什么职责分流**"。

分布式系统（SDN、数据库、K8s、服务网格）几十年的共识是另一条正交切分：
- **数据面（Data Plane）**：真正搬运/处理业务负载的高频路径。
- **控制面（Control Plane）**：决定数据面"如何搬运"的低频决策路径。

把这条切分引入 agentic runtime，有三个直接收益：

1. **科学性**：借用成熟系统理论的术语与不变量（control/data plane 分离是可验证的工程定律），直接削弱"工程而非科学"的质疑——我们不是发明分层，而是论证 agent 系统**满足**经典正交切分，并给出它在概率性 LLM 调用下的特殊形态。
2. **P1 第二层机制解释**：逻辑扩展主要改变**控制面决策空间**，物理扩展主要改变**数据面吞吐容量**；二者作用在不同平面 → 解耦有了结构性原因（不止"契约层 H"这一个理由）。
3. **维护子系统 $M$ 定位精确化**：[03-Harness-System-SubAgents清单](03-Harness-System-SubAgents%E6%B8%85%E5%8D%95.md) 的 $M$ 一直"横跨"逻辑/物理轴，难以归位。用 control/data plane 一拆即清：**M 的触发决策在控制面，M 的 token 开销在数据面**。

---

## 1. 定义

沿用 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) 的 $A = \langle S, H, X \rangle$。在其上叠加一个**平面投影** $\pi: A \to \{\text{DP}, \text{CP}\}$，把系统的每个职责单元映射到数据面或控制面。

**定义 3（数据面 Data Plane, DP）.**
承载**单位请求的 token 处理路径**：Skills 组装的 context → LLM 推理 → 工具执行 → 结果返回。其负载与 **token 流量成正比**，是 throughput $\Theta$ 的物理载体。
$$\text{DP} = \{\text{context 组装执行}, \text{LLM 前向}, \text{工具调用执行}, \text{维护推理的 token 开销}\}$$

**定义 4（控制面 Control Plane, CP）.**
承载**决策与策略**：决定数据面"如何流动"，但本身不搬运业务 token（或搬运量与业务流量**解耦**）。
$$\text{CP} = \{\text{契约翻译规则}\ H{:}\mathcal{I}\to\mathcal{E},\ \text{路由},\ M\ \text{触发策略}\ (\theta_{compact},\theta_{persist}),\ \text{不变量执行}\ \text{Inv},\ \text{生命周期编排}\}$$

**平面分离公理（Plane Separation）.**
良构的 agentic runtime 满足：控制面决策的产生频率 $f_{CP}$ 与业务 token 流量 $\dot{\tau}$ **不成正比**（$f_{CP}$ 由状态事件触发：context 超阈、step 边界、skill 集合变更），而数据面负载 $L_{DP} \propto \dot{\tau}$。

---

## 2. 与扩展轴的二维正交框架

两条切分相互正交，构成 2×N 的设计矩阵：

| | **数据面 DP**（高频，∝ token 流量） | **控制面 CP**（低频，∝ 状态事件） |
|---|---|---|
| **L3 Skills** | task skill 被激活后产生的 context tokens | task skill 的**选择/调用/终止/循环条件**判定 |
| **L2 Harness** | 工具执行的 token 流、$M$ 维护推理的 token 开销 | 契约翻译规则、路由、$M$ 的触发策略、不变量执行 |
| **L1 Scaffold** | sandbox 内实际计算、serving 前向 | 实例调度、放置、隔离边界设定 |

> **关键观察**：扩展轴（逻辑/物理）问"加什么"，平面轴（CP/DP）问"按什么职责分流"。一个职责单元同时有扩展归属和平面归属——例如"维护子系统 $M$"：扩展归属=物理轴（见 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) §1.1 P1），平面归属=**触发在 CP、开销在 DP**。

## 3. 命题扩展

### 命题 P4（平面分离 Plane Separation）

> **在良定义契约 $H$ 下，控制面负载与数据面负载相互独立：$\dfrac{\partial L_{CP}}{\partial \dot\tau} \approx 0$ 且 $\dfrac{\partial L_{DP}}{\partial |\text{CP 规则}|} \approx 0$。**

即：业务流量增大不会成比例放大控制决策开销（控制面可被薄薄一层承载、集中式或低频）；控制规则变复杂（如新增 skill 调用条件、新增不变量）不会成比例放大每请求的数据面 token 成本。

**实验可证伪形式**：
- 固定控制规则集，令业务 QPS $\uparrow$ → 测控制面 CPU/调度开销占比 → 期望占比**不随 QPS 上升**（数据面线性、控制面亚线性）。
- 固定 QPS，令控制规则数（skill 条件 + 不变量）$\uparrow$ → 测单请求 token 成本 → 期望**基本不变**；若显著上升 → 控制逻辑泄漏进了数据面（反模式），平面分离被破坏。

### 命题 P5（P1 的平面分解 / P1 经由平面解释）

> **逻辑扩展（+task skill）的一阶效应落在控制面（扩大 CP 决策空间），物理扩展（+Scaffold）的一阶效应落在数据面（扩大 DP 吞吐容量）；因二者作用平面不同，故在效果空间 $(\mathcal{T},\Theta)$ 上正交——这是 P1 的结构性来源。**

形式化承接 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) P1：
$$
\underbrace{+s \Rightarrow \Delta\text{CP 决策空间}}_{\text{逻辑扩展}\to\text{控制面}}
\qquad
\underbrace{+x \Rightarrow \Delta\text{DP 吞吐容量}}_{\text{物理扩展}\to\text{数据面}}
$$
P1 的反例风险（"加 Skills 撑大 context 拖垮 throughput"）在此被精确定位为**控制面逻辑泄漏到数据面**：若 skill 的全量 schema 每请求都进 context（数据面），则逻辑扩展污染了数据面。**缓解 = 把 skill 规约留在控制面，仅按需把激活路径注入数据面**——这正是 Tool Forge intent-scoped 加载省 99.2% context（[精读-Tool-Forge-2605.28000](%E7%B2%BE%E8%AF%BB-Tool-Forge-2605.28000.md)）的平面学解释。

### 命题 P6（维护子系统的跨面分解）

> **维护子系统 $M$ 的触发逻辑属控制面（由 $\theta$ 阈值与状态事件驱动，频率与业务流量解耦），其推理开销属数据面（消耗 serving token，随物理吞吐线性）。** 故 $M$ 不破坏平面分离，反而是平面分离的典型实例。

## 4. 与现有系统理论的对照（写进 related work）

| 领域 | data plane | control plane | 本架构对应 |
|---|---|---|---|
| SDN | 转发分组（line-rate） | 路由/流表下发 | DP=token 推理流；CP=契约+路由+$M$ 触发 |
| K8s | Pod 内业务流量 | API server / scheduler | DP=Scaffold 内执行；CP=实例调度/放置 |
| 数据库 | 查询执行算子 | query planner / optimizer | DP=工具执行；CP=契约翻译 $H{:}\mathcal I\to\mathcal E$（≈ planner）|
| 服务网格 | sidecar 代理流量 | xDS 配置分发 | DP=工具调用流；CP=能力供给与路由策略 |
| **agentic runtime（本架构）** | **token 处理路径** | **决策/翻译/维护触发/不变量** | —— |

> **novelty 声明（须在论文强调）**：经典系统的控制面是**确定性配置下发**；本架构的控制面**部分由 LLM 驱动**（契约翻译、$M$ 反思、tool synthesis 都可能调模型），是"**概率性控制面（probabilistic control plane）**"。这带来经典理论没有的问题：控制决策本身有不确定性，需用**可审计日志 + 不变量执行**把概率性控制面约束在安全包络内（呼应 ClawVM 的 auditable，见 [03-Harness-System-SubAgents清单](03-Harness-System-SubAgents%E6%B8%85%E5%8D%95.md)）。这一差异是本平面切分的真正贡献点，而非简单套用 SDN。

## 5. 对 C5 / 主线的影响

1. **C5 从一维升级为二维**：逻辑/物理扩展（加什么）× 控制/数据面（怎么分流）→ 2×3 设计矩阵，每格可独立优化 NFR（衔接 C6）。
2. **P1 获得第二独立论据**：除"契约层 H"（P2）外，"作用平面不同"（P5）是解耦的第二个结构性原因 → 论文论证更稳健。
3. **维护子系统归位**：P6 用跨面分解一举消除"$M$ 到底算逻辑还是物理"的含混。
4. **科学化弹药 +1**：可借用 control/data plane 分离这一成熟工程定律，并指出"概率性控制面"是 LLM agent 特有的新问题。

## 6. 待验证 / 风险点

- [ ] **控制面是否真能保持低频？** tool synthesis、反思都调 LLM，若高频触发，CP 自身吃大量推理 → 退化为"CP 也在数据面上"。需界定"控制决策频率上界"假设（与 $\theta$ 阈值设计相关，见 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) §S5）。
- [ ] **P4 的 $\partial L_{CP}/\partial\dot\tau\approx0$ 在集中式控制面下是否成为瓶颈？** 借鉴 SDN 的控制面水平扩展经验。
- [ ] **概率性控制面的正确性如何度量？** 需要可审计性指标（决策可重现率 / 不变量违例率）。
- [ ] 平面投影 $\pi$ 是否对所有职责单元都良定义？是否存在"既不纯 CP 也不纯 DP"的灰色单元（如 streaming 场景的增量决策）。
