# 10 — Skill-as-Code 与确定性固化（Determinism Anchoring）

> **来源**: 2026-07-12 研究讨论。承 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) §0.5 的机制根因（推理=内在知识⊕context）与 [08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) §2 的 N0 精化（概率决策器 ⊕ 确定护栏门）。
> **类型**: 论文核心理论扩展 —— 给「概率性控制面 N0」补上其一直缺失的另一半：**确定性从哪里来**。
> **状态**: 草稿 v0.1
> **定位**: 前四维回答「怎么解耦扩展 / 怎么在预算内并行」；本文回答**在一个由 LLM 驱动的概率控制面里，如何让高确定性需求的执行段落变得确定、可复现、且抗模型版本漂移**。核心手段：当一个 skill 的操作空间基本闭合时，把其「获取→判断→核验→决策」过程固化为 **code**，使这段推理从"每轮概率生成"退化为"确定执行"。

---

## 0. 一句话总纲

> 一次推理的输出 = 模型内在知识（权重 θ，可变且随版本演进）⊕ context（可变）。对确定性需求高、操作空间闭合的 skill，**用 code 表达其过程**，就把这段输出从"依赖 θ 的概率生成"移出到"不依赖 θ 的确定执行"。**code 不只是提效手段，更是概率控制面里的确定性来源，也是模型版本演进下的契约稳定锚。** Harness 的一项核心职责，就是**总结 skill 并尝试把其可固化过程记录为 code**，同时在 context 中保留一定比例的自然语言（NL），用于多样性校验与 reflection 的随机反例——形成一条沿 agent loop 阶段变化的**确定性梯度**。

---

## 1. 机制根因：为什么"闭合 → 可 code 化 → 确定"

承 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) §0.5 的第一性原理：**agent 唯一能动的变量是 context**；模型权重 θ 在一次任务内不变、但**跨版本会变**。据此：

**(1) 确定性来源的分层**。一段执行的不确定性有两个源头——① θ 的概率采样（同 context 也可能输出不同）；② context 组织的不稳定。当 skill 满足：
- **工具集闭合**：$\tau(\text{skill})$ 在生命周期内基本不跳变（接 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) M1/P11）；
- **输出格式固定**：$\sigma^{out}$ 是结构化的（表格 / bullets / schema）；
- **判断规则可枚举**：获取→判断→核验→决策的分支基本闭合；

则这段过程**可被 code 表达**。一旦 code 化，① θ 采样被移除（code 是确定函数），② context 组织被冻结进代码逻辑。**两个不确定性源头同时被消掉。**

**(2) 与 P11 的对偶关系**。P11（前缀冻结）冻结的是 **context 前缀**，省的是 KV 重算；skill-as-code 冻结的是**决策过程本身**，省的是**整段概率生成**。二者是同一根"冻结稳定部分"杠杆在不同层面的投影：

| | 冻结对象 | 收益 | 命题 |
|---|---|---|---|
| P11 | context 前缀（逐 token） | KV 复用（快）+ 注意力不稀释（准） | 局部性 |
| **本文** | **决策过程（整段）** | **免概率生成（确定 + 快）+ 抗 θ 漂移** | **N8/P14** |

> **可 code 化不是新判据，而是工具集闭合度的阈值现象**：当 $\rho\to 0$、$\tau$ 闭合、$\sigma^{out}$ 固定到某阈值，skill 从"需 LLM 概率生成"退化为"可 code 确定执行"。这把 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) 的连续量（$\rho, H$）接上了一个离散的相变。

## 2. N0 的三分：概率控制面里的确定性锚

[08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) §2 已把控制面精化为二分：**概率决策器 ⊕ 确定护栏门**。但那里的"确定护栏门"只回答了**校验确定**（Inv/权限/审计），没回答**生成确定**从哪来。skill-as-code 补上第三层：

| 子层 | 机制 | 确定性来源 | 关联 |
|---|---|---|---|
| 概率**决策** | LLM 生成计划 / 子 agent 切分 / 反思 | 无（本体概率，N0） | [05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md) |
| **确定执行**（★ 本文新增） | **skill-as-code：闭合操作空间的过程固化** | **code 本身**（确定函数） | N8 |
| 确定**护栏** | Inv / 权限 / 审计门 | 校验规则 | [08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) §2、A6 |

> **精化后的 N0 表述**：控制平面 = **概率决策器 ⊕ 确定执行体 ⊕ 确定护栏门**。其中"确定执行体"就是被 code 固化的 skill 段落——它把 A3（概率执行公理）的作用域从"整个控制面"收窄到"未 code 化的决策部分"，让委派率（[08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) §6 协作悖论：委派率仅 0–20%）有了除"护栏"之外的第二个抓手：**可 code 化的部分先确定下来，人只需审概率部分。**

## 3. 形式化与命题（N8 / P14）

沿用 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) 的 $A=\langle S,H,X\rangle$ 与 skill 形式 $s=\langle\text{desc},\sigma^{in},\sigma^{out},c\rangle$。引入：

- **可固化度** $\kappa(s)\in[0,1]$：skill $s$ 的过程可被确定 code 表达的比例（工具集闭合度 × 输出结构化度 × 分支可枚举度的合成）。
- **code-frozen skill** $s^{\text{code}}$：$\kappa(s)$ 超阈后，其过程被 Harness 固化为 code 的形态；执行时不调用 θ（或仅在留白点调用）。
- **输出方差** $\text{Var}_\theta(s)$：固定 context、跨模型版本 $\theta_t\to\theta_{t+1}$ 重跑时输出的分布方差。

### 命题 P14（跨模型稳定性 / Model-Version Stability）— 本文主命题

> **被 code 固化的 skill 段落，其输出分布不随模型版本演进而漂移；只有未 code 化的概率段落随 θ 变化。** 形式化：
> $$\frac{\partial\,\text{output}}{\partial\theta}\Big|_{s^{\text{code}}}\approx 0,\qquad \text{Var}_\theta(s^{\text{code}}) \ll \text{Var}_\theta(s^{\text{LLM}})$$
> 即 code 化是 [01-C5-双扩展解耦形式化与命题](01-C5-%E5%8F%8C%E6%89%A9%E5%B1%95%E8%A7%A3%E8%80%A6%E5%BD%A2%E5%BC%8F%E5%8C%96%E4%B8%8E%E5%91%BD%E9%A2%98.md) P2（契约充分性）在**时间维**的扩展：P2 让 S 与 X 空间解耦，P14 让 skill 输出与 θ **版本解耦**。

**可证伪**：取同一 use case，准备两种实现——(a) 纯 LLM skill；(b) code-frozen skill。在 $\theta_t, \theta_{t+1}, \theta_{t+2}$ 三个模型版本上各重跑 N 次，测输出方差 / 格式一致率 / 结果可复现率。P14 预期 (b) 的跨版本方差显著低于 (a)；若 (b)≈(a) → code 化对模型漂移无稳定作用，P14 被证伪。

### 命题 N8 的可证伪投影（确定性提升）

> **对 $\kappa$ 高的 skill，code 化在固定 θ 下也降低同 context 的输出方差、提升执行效率，且该 code 可被复制进工具集作为其它 agent 的输入（capability capsule）。**

**可证伪**：同一 θ、同一 context，比较纯 LLM skill 与 code-frozen skill 的 (i) 输出方差、(ii) 单次延迟/token 成本、(iii) 被其它 agent 复用后的成功率。预期 code 化显著降方差、降成本、可复用；若无差异 → 操作空间未真正闭合（$\kappa$ 估计偏高），需回退 §1 的闭合判据。

## 4. 设计律：确定性梯度（沿 agent loop 的 code:NL 配比）

你不能把整个 agent 都 code 化——那会杀死泛化。关键是**分阶段**：

| loop 阶段 | 确定性需求 | code : NL 配比 | 理由 |
|---|---|---|---|
| **子 agent 派生 / 执行** | **高** | 偏 code | 操作空间闭合、需可复现、抗漂移（P14）；接 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) M1/M4 |
| 取数 / 核验 | 中高 | code 为主 + NL 兜底 | 规则可枚举，但边界情形留 NL |
| **全局 reflection / 内容校验** | **低（需高多样性）** | 偏概率 NL | 需 θ 内在知识做泛化判断 + 随机反例增强 |

**保留 NL 的两个作用**（务必写进 paper，防"全 code 化"误读）：
1. **多样性校验**：同一结论用 NL 概率生成一份对照，检出 code 分支未覆盖的情形（利用 θ 内在知识，呼应 [08-模型原生计算架构-体系结构视角](08-%E6%A8%A1%E5%9E%8B%E5%8E%9F%E7%94%9F%E8%AE%A1%E7%AE%97%E6%9E%B6%E6%9E%84-%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84%E8%A7%86%E8%A7%92.md) A3"统计质量边界 + 语义验收"）。
2. **reflection 的随机反例 shots**：在 context 的 few-shot 里随机插入反例，增强 agent 反思时的泛化能力，防止 code 化把 agent 锁死在已知分支。

> **设计律（一句话）**：**确定性沿 loop 阶段是梯度而非开关**——派生/执行端向 code 收敛（要可复现），reflection/校验端向 NL 发散（要泛化）。code:NL 配比本身是 Harness 的一个可调设计变量。

## 5. Harness 的新职责：skill 总结与 code 沉淀

这把 [03-Harness-System-SubAgents清单](03-Harness-System-SubAgents%E6%B8%85%E5%8D%95.md) 的维护子系统 $M$ 扩了一条能力：

```
summarize_and_freeze(skill_trace) -> s_code | s_llm
  观察 skill 的历史执行轨迹（工具集/分支/输出格式）
  估计 κ(skill)  = closure(τ) × struct(σ_out) × enumerability(branches)
  if κ ≥ θ_freeze:
      提炼过程 → code（确定函数）+ 契约（intent/输入/输出/验证/生命周期）  # Tool-Forge capsule 形态
      注册进工具集，供其它 agent 复用                                      # 可复制性
  else:
      保留为 LLM skill，记 NL few-shot（含随机反例）
  始终在 context 保留 code:NL 配比（§4 梯度）
```

- **与 tool synthesis 的关系**：这不是又造一个新工具，而是**把已被验证稳定的 skill 过程"降格"为 code**——方向相反：tool synthesis 是"生成新能力"，skill-as-code 是"固化已稳定能力"。二者都归 Harness 能力供给（[03-Harness-System-SubAgents清单](03-Harness-System-SubAgents%E6%B8%85%E5%8D%95.md) E）。
- **与 D₃ data-usage skill 的关系**：D₃（[06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md)）把"如何用数据"沉淀为 skill；本文把"如何执行闭合过程"沉淀为 code。二者同构——**都是把重复出现的稳定决策从概率生成中固化出来**（D₃ 沉淀取数决策，本文沉淀执行过程）。P9（治理记忆收敛）与 P14（跨模型稳定）是同一"沉淀降熵"机制的两个面。

## 6. 对主线 / novelty 的影响

1. **N8（拟）· skill-as-code 作为确定性锚**：把 N0 的二分（概率决策 ⊕ 确定护栏）升级为三分（+ **确定执行体**），回答了"概率控制面里确定性从哪来"。对标空白：CodeAct / AutoHarness / Tool-Forge 讲"code 作为 action 提效/可组合"，**无人把 code 定位为"概率控制面的确定性锚 + 模型版本演进下的契约稳定器"**。
2. **P14（拟）· 跨模型稳定性**：P2 契约充分性的**时间维扩展**——被 code 固化的 skill 输出与 θ 版本解耦。这是模型迭代极快的现实下，一个此前未被形式化的稳定性主张。
3. **确定性梯度设计律**（§4）：给 [09-并行度与局部性协同设计](09-%E5%B9%B6%E8%A1%8C%E5%BA%A6%E4%B8%8E%E5%B1%80%E9%83%A8%E6%80%A7%E5%8D%8F%E5%90%8C%E8%AE%BE%E8%AE%A1.md) §7 风险"概率性划分的稳定性"一个正面答案——派生端用 code 稳住，reflection 端用 NL 保泛化。
4. **skill 理论第四形态**（备选定位，本轮不采）：在 task/system/data-usage 三分（N5）之外，code-frozen 是**正交于"服务对象"的"表示形态"维度**——同一 skill 可以是 LLM 形态或 code 形态。本轮按用户决定采「确定性锚」主定位，此形态视角作为 §7 备注保留。

## 8. 待验证 / 风险点

- [ ] **$\kappa$ 的可操作估计**：closure(τ)×struct(σ_out)×enumerability 三因子如何量化？需要一个可计算的 freeze 判据，否则退回工程直觉。
- [ ] **过早 freeze 的脆性**：若在操作空间尚未真正闭合时 code 化，边界情形会硬失败（不像 LLM 能兜底）→ 需 §4 的 NL 兜底层 + 回退机制。
- [ ] **P14 的实验成本**：需跨 ≥3 个真实模型版本重跑，获取多版本可能受限；可用同族不同 size 或不同温度近似。
- [ ] **code 与 NL 一致性漂移**：code 分支与 NL 校验若长期分叉，谁为准？需定义 code 为 ground truth、NL 为 diversity probe 的仲裁规则。
- [ ] **可复制性的权限边界**：code 化 skill 复制进工具集供其它 agent 用时，须继承 P3 组合不变量（防越权组合，[精读-Benign-in-Isolation-技能组合风险-2606.15242](%E7%B2%BE%E8%AF%BB-Benign-in-Isolation-%E6%8A%80%E8%83%BD%E7%BB%84%E5%90%88%E9%A3%8E%E9%99%A9-2606.15242.md)）。
