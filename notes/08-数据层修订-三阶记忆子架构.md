# 08 — 数据层修订：三阶记忆子架构（Data Layer v2.0）

> **来源**: 2026-07-01 研究讨论，对 [06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md) 的系统性修订，映射用户提出的“Raw Data → Index & Link → Data Theme”三阶记忆空间
> **类型**: 论文核心理论扩展（formalization extension / architecture revision）
> **状态**: 草稿 v0.1
> **定位**: 将原数据子系统 𝒟 = ⟨D₁, D₂, D₃, D₄, Ω⟩ 的“接口/平面”视角，升级为**企业级三阶记忆子架构**——自下而上（Raw Data → Index）与自上而下（Data Theme）双螺旋，融入 5W1H+Which 叙事本体、Karpathy LLM Wiki 模式、以及企业 Lakehouse 语义层治理实践。

---

## 0. 修订动机：为什么从“四接口”升级为“三阶记忆”

[06-数据平面四组成架构](06-%E6%95%B0%E6%8D%AE%E5%B9%B3%E9%9D%A2%E5%9B%9B%E7%BB%84%E6%88%90%E6%9E%B6%E6%9E%84.md) 用 D₁–D₄+Ω 定义了 agent 与外部数据交互的**接口契约**，在 Harness 层面完成了“取数-总结-治理-生命周期”的形式化。但它在企业部署中暴露出一个结构性缺口：

> **四接口回答的是“agent 怎么访问数据”，但没有回答“企业数据怎么被 agent 理解、记忆、复用”**。当多个 agent、多个 use case、多个租户共享同一数据生态时，需要一个**持久化、可导航、可沉淀的 memory substrate**——而不仅是瞬时的 API 调用。

本次修订引入**三阶记忆空间**（Three-Tier Memory Space），把四接口重铸为：

| 三阶记忆 | 核心问题 | 原接口映射 | 类比（经典系统）|
|---------|---------|-----------|--------------|
| **Tier 1 · Raw Data** | 数据在哪、以什么形式存在 | D₁ 的底层存储 + D₄ 的物理元数据 | 数据库的 page / 文件系统的 inode |
| **Tier 2 · Index & Link** | 数据是什么、怎么找、和谁有关 | D₂ 的 off-policy 语义总结 + 5W1H+Which 本体 | 数据库的索引 + 搜索引擎的倒排 + 知识图谱 |
| **Tier 3 · Data Theme** | 数据被怎么用、为什么用、用出了什么经验 | D₃ 的 governance memory + D₄ 的时效/NFR 策略 | 数据仓库的语义层 + BI 的 metric 定义 + 组织过程资产 |

> **关键洞察**：原 D₁–D₄ 是**功能视角**（接口做什么），三阶记忆是**存储视角**（memory 如何分层沉淀）。二者正交：同一套 D₁–D₄ 接口可以在三阶记忆的任何一层被调用，但每层对数据的**理解深度**和**复用半径**不同。

## 1. Tier 1 · Raw Data — 原始数据湖

### 1.1 定义与范围

Raw Data 是企业数据的**物理着陆层**（physical landing layer），对接云上 Lakehouse（或混合部署的本地存储），存放一切未经语义加工的原始信息：

- **原始文件**：PDF 报告、Excel 表格、Word 文档、扫描件、图片、音视频转录
- **数据库连接/视图**：ERP / SCM / CRM / SAP / 飞书多维表格 的实时连接、只读视图、CDC 流
- **报表与 Dashboard 快照**：BI 系统定时导出的指标快照（如 Tableau / Power BI 的 PDF/CSV 导出）
- **Agent 推理轨迹**：agent 自身的 thinking chain、tool call 日志、反思记录、中间产物（code / SQL / 草稿）

> **核心原则**：Raw Data 层**不解释数据含义**，只保证**可用性、完整性、可审计性**。schema-on-read 是默认假设——原始信息的潜在结构（latent schema）被完整保留，不因人为建模而截断。

### 1.2 与 Lakehouse 的对接

企业级 Raw Data 层直接映射到现代 **Data Lakehouse** 架构（Apache Iceberg / Delta Lake / Hudi）：

| Lakehouse 组件 | Raw Data 层对应 | 功能 |
|---------------|----------------|------|
| Object Store（S3 / OSS / GCS） | 原始文件仓 | 低成本、高吞吐的冷/热分层存储 |
| Table Format（Iceberg / Delta） | 结构化数据视图 | ACID 事务、时间旅行、分区裁剪 |
| Catalog（Polaris / Hive / Unity） | 数据源注册表 | 统一命名空间、跨引擎访问 |
| Streaming Ingestion（Kafka / Flink） | CDC 与事件流 | 实时数据入湖、变更捕获 |

> **关键设计**：Raw Data 层对 agent 暴露的**不是物理路径**，而是**Catalog 中的逻辑注册项**（`src_id`）。这使 agent 在权限、审计、生命周期上与企业数据治理体系对齐，而不是绕过它。

### 1.3 与既有研究的关联

- **Karpathy LLM Wiki 的 `raw/` 层**：Karpathy 在个人知识管理场景中定义了 `raw/` 目录作为“immutable source dump”。企业级 Raw Data 层是其工业规模放大——保留“原始信息不可变”原则，但引入 Lakehouse 的 ACID、分区、Catalog 治理。
- **Halevy 等 (2605.28787) *Do Agents Need Semantic Metadata?***：该实证研究表明“裸 agent 在线发现原始数据”覆盖高但精度低。Raw Data 层作为裸发现的**物理基础**，但本身不提供语义——语义交给 Tier 2 解决。
- **命题 P7（取数解耦）的根基**：∂ℓ/∂|src| ≈ 0 成立的前提是 Raw Data 层的规模增长**不直接涌入 agent context**。Tier 2 的索引隔离正是 P7 的机制保障。

## 2. Tier 2 · Index & Link — 语义索引与事实链

### 2.1 定义：自底向上的总结

Tier 2 是 Raw Data 的**语义编译层**。它由一组 **off-policy 离线 agent 集群**（Off-Policy Indexing Cluster, OPIC）持续运行，不断读取 Raw Data，生成**结构化事实（Facts）**并建立**交叉链接**。这些 agent 不在主请求路径上，而是独立 loop，类比搜索引擎的爬虫与索引构建。

> **核心原则**：Index & Link 是**自底向上**（bottom-up）的——从 Raw Data 的原始内容中提炼意义，而不是从预设的 schema 中向下填充。

### 2.2 Facts 的 5W1H+Which 叙事本体

每个 fact 是一个**叙事性描述**，按 5W1H+Which 七维组织，形成统一的知识原子：

| 维度 | 字段 | 说明 | 示例（RDBMS table） | 示例（会议记录） |
|------|------|------|-------------------|----------------|
| **What** | `fact_type` | 这是什么：人物、事件、报告、会议梗概、工作计划、代码、工具、数据表元数据 | `TABLE: sales_order` | `MEETING: 产品需求评审` |
| **Who** | `producers`, `consumers`, `excluded` | 谁产生的、给谁看、不给谁看的 | 产生：销售系统；消费者：财务/运营；排除：外部供应商 | 产生：产品经理 Alice；消费者：研发团队；排除：销售团队 |
| **Where** | `raw_sources`, `upstream_indices`, `access_boundary` | 来自哪些 Raw Data 或其它 Index；数据访问边界：互联网、内网、专用应用（SAP、飞书） | 来源：`erp.sales_order` 表；边界：内网 + SAP 专用网关 | 来源：飞书妙记录屏 + 纪要文档；边界：内网飞书 |
| **When** | `created_at`, `modified_at`, `valid_until`, `obsoleted_at` | 产生、修改、作废时间线；实效性窗口 | 创建于 2023-01；最新分区 2026-06-30；口径 2024-Q2 变更 | 2026-06-28 14:00–16:00；纪要整理于次日 10:00 |
| **Why** | `causality_chain`, `triggered_by`, `next_steps`, `later_outcomes` | 前因后果：产生上下文、触发原因、后续动作、结果反馈 | 触发：Q2 财报需求；后续：生成 `revenue_summary` 报表；结果：发现华东区口径不一致 | 触发：用户投诉响应延迟；后续：创建 #123 工单；结果：已修复并于 7/1 上线 |
| **How** | `content_digest`, `key_list`, `short_narrative` | 展开内容：对简单 facts 的列表，或对复杂 facts 的短叙事 | 字段列表：order_id, customer_id, amount, region, status；行数：1.2M；分区：按月/region | 决议：① 增加埋点 ② 缩短 SLA 至 4h ③ 责任人 Bob；遗留问题：数据回传延迟未解决 |
| **Which** | `temporal_links`, `cross_refs` | 在特定时效段内，该 fact 的关联关系，用于交叉查找 | 关联：→ `customer_dim` (FK) → `region_dim` (FK) → 华东区口径变更事件 | 关联：→ #123 工单 → 6/15 同类会议 → 产品路线图 Q3 |

> **设计来源**：5W1H 是新闻学、情报学和事件本体论的经典框架。Piryani 等 (2024) 的 **XPEventCore 本体**首次将 5W1H 整合为事件知识图谱（EKG）的完整表示，并证明其在案例推理（EvCBR）上的显著增益。本架构将 XPEventCore 从“事件”泛化为“一切数据单元的事实描述”，并增加 **Which** 维度以显式记录关联导航——这是对 XPEventCore 的 enterprise 扩展。

> **与 SoKG (2601.10003v2) 的呼应**：SoKG 提出“5W1H-guided QA generation to systematically construct KGs from source documents”，其消融实验表明 5W1H 引导的问答生成比 generic QA 在 KG 构建的 factual retention 和 structural cohesion 上均有显著提升。Tier 2 的 OPIC 集群本质上是在执行一种**持续的、企业级的 SoKG 构建**——只不过产物不是静态 KG，而是活的、带时效的、可交叉引用的 fact graph。

### 2.3 对不同 Raw Data 类型的索引策略

| Raw Data 类型 | OPIC 行为 | Fact 重点 | 索引方式 |
|--------------|----------|----------|---------|
| 文件（PDF/Word/Excel） | 提取文本/表格/图表 → 按段落/表/页生成 facts | 内容摘要、关键数据点、结论 | 全文检索 + 实体链接 |
| RDBMS 表/视图 | 读取 schema + 采样数据 → 生成 metadata + 样本 profile | 表用途、字段口径、关联表、数据分布 | Schema 索引 + 数据血缘图 |
| 报表/Dashboard | 解析结构（标题/指标/维度） → 提取指标定义 | 指标名称、计算公式、时间粒度、对比维度 | Metric 语义索引 |
| Agent 推理轨迹 | 压缩 thinking chain → 提取决策树 + 反思点 | 任务目标、工具选择、错误与修正、最终结论 | 轨迹检索 + 经验模式挖掘 |
| API/Stream 数据 | 采样窗口 + 结构推断 → 生成动态 schema | 接口用途、字段类型、频率、异常模式 | 动态 schema 注册 + 流特征索引 |

### 2.4 链接（Link）机制：Which 维度的实现

Which 维度不是简单的“相关推荐”，而是**基于 5W1H 其余六维的关联推断**：

- **时间关联**：同一 `when` 窗口内发生的事实自动建立**共时链接**（synchronous link）
- **因果关联**：`why` 中的 `triggered_by` 和 `next_steps` 建立**因果链**（causal link）
- **语义关联**：`what` 的 type 和 `how` 的 content 经 embedding 建立**相似链接**（semantic link）
- **访问关联**：`where` 的 `access_boundary` 相同的事实建立**域内链接**（domain link）
- **血缘关联**：数据表之间的 FK/PK/ETL 依赖建立**血缘链接**（lineage link）

> **与 Zep/Graphiti 的关联**：Zep (2025) 提出 Temporal Knowledge Graph 的三层结构（episode → semantic entity → community），其中 episode 对应 Raw Data，semantic entity 对应 Tier 2 的 fact 节点，community 对应 Which 的跨域链接。本架构的 Which 维度是 Zep 的 temporal/semantic/entity 链接的**企业级扩展**，增加了因果链、访问域、数据血缘等工程维度。

### 2.5 平面归属与独立性

Tier 2 的生产侧（OPIC 集群）是**完全独立的 off-policy loop**，不在三层栈 ⟨S, H, X⟩ 内：

- **自己的循环**：`observe(Raw Data 变更) → extract(5W1H) → link(Which) → write(Index) → 触发增量更新`
- **自己的 sub-agents**：文档提取 agent、schema 探查 agent、关联推断 agent、质量校验 agent
- **甚至自己的模型**：索引模型可与主 agent 不一致（专用 summarizer / embedding / 小模型）

> **原 D₂ 的精确对应**：Tier 2 就是 D₂ 的**语义总结平面**的 concrete instantiation——从抽象的“off-policy lake”具象为“5W1H+Which 结构化 fact 库”。其**查询接口**（semantic join）仍然是 Harness 直接消费的部分，但生产 loop 的独立性不变。

## 3. Tier 3 · Data Theme — 数据主题与经验记忆

### 3.1 定义：自顶向下的总结

如果说 Tier 2 是“数据告诉你它是什么”，Tier 3 就是“组织告诉数据它意味着什么”。Data Theme 是**自顶向下**（top-down）的总结与记忆——记录“什么时候、什么应用、在什么上下文中来找过这些数据、找了哪些、怎么导航的、最后输出了什么”。

> **核心原则**：Data Theme 把“数据使用行为”本身固化为**可复用的组织记忆**，避免每次 use case 都重新探索数据。它是企业知识管理的“最后一公里”。

### 3.2 Data Theme 的 5W1H 记录结构

每个 Data Theme 记录一次（或一类）数据访问的完整上下文，同样按 5W1H 组织：

| 维度 | 字段 | 说明 | 示例 |
|------|------|------|------|
| **What** | `theme_name`, `intent_type`, `output_spec` | 主题名称、检索意图类型、预期输出规格 | `theme`: Q2 华东区营收分析；`intent`: 多表关联 + 时序对比；`output`: 带图表的 Markdown 报告 |
| **Who** | `requester_agent`, `requester_user`, `target_audience` | 哪个 agent 发起、哪个用户授权、结果给谁 | Agent: `financial-analyst-v3`; User: 财务总监; Audience: 管理层周报 |
| **Where** | `data_sources_used`, `index_nav_path`, `tools_invoked` | 实际使用了哪些数据源、索引导航路径、调用了哪些工具 | 源：`sales_order` (D₁) + `revenue_summary` (Tier 2 fact)；路径：从“华东区”region_dim → 时间筛选 → 关联订单表；工具：SQL 生成、PyPlot 图表、飞书推送 |
| **When** | `query_time`, `data_as_of`, `result_freshness_req` | 查询发生时间、数据截止时点、结果新鲜度要求 | 2026-06-30 09:00；数据截止 2026-06-28；要求 T+1 |
| **Why** | `business_question`, `decision_context`, `action_taken` | 业务问题、决策背景、后续行动 | 问题：Q2 华东是否达标？背景：季度经营分析会；行动：若未达标则触发 #456 成本审计 |
| **How** | `retrieval_strategy`, `join_logic`, `fallback_behavior` | 检索策略、关联逻辑、失败回退行为 | 策略：先查 Tier 2 Index 定位口径，再经 D₁ 拉原始明细；关联：sales_order ⋈ region_dim ⋈ product_dim；回退：若华东口径 2024 后变更，则自动切换至新口径视图 |

> **与 D₃ 的精确对应**：Tier 3 就是 D₃（data-governance memory）的**结构化扩展**。原 D₃ 定义为“per-use-case 的数据使用 skills 集合”，而 Data Theme 把这些 skills 的**实例化记录**固化为持久 memory。二者关系：D₃ 是**类**（class），Data Theme 是**对象**（instance）——一次成功的数据访问被沉淀为 Theme，下次同类 use case 直接复用 Theme 的 5W1H 记录，而非重新探索。

### 3.3 Theme 的复用与收敛机制

Data Theme 的核心价值在于**复用**：

1. **Theme 匹配**：当新请求到来时，先查 Tier 3 的 Theme 库，按 `what` + `who` + `where` 的相似度匹配历史 Theme。
2. **Theme 继承**：若匹配到近似 Theme，则**继承**其 `how`（检索策略、关联逻辑）和 `where`（数据导航路径），只需微调 `when` 和 `why`。
3. **Theme 演化**：若执行过程中发现新数据源或新关联方式，则**更新** Theme 的 5W1H 记录，并标记版本。
4. **Theme 废弃**：当 `when` 中的数据口径失效、或 `where` 中的数据源下线时，Theme 被标记为 `stale`，触发 OPIC 重新索引或人工复核。

> **与 P9（治理记忆收敛）的对应**：重复运行同类 use case 的取数决策熵 ℋ(π_data⁽ⁿ⁾) 单调不增——这正是 Theme 复用机制的形式化表达。每一次 Theme 命中，agent 的“探索式取数”就转化为“利用式取数”，决策方差下降。

### 3.4 与 Enterprise Semantic Layer 的融合

企业级 Data Theme 直接对接现代 Lakehouse 的 **Semantic Layer**（Dremio / dbt / Cube / Atlan）：

| Semantic Layer 组件 | Data Theme 层对应 | 功能 |
|--------------------|------------------|------|
| Metrics Store / Metric Layer | Theme 的 `what.output_spec` | 统一指标定义（口径、粒度、计算公式） |
| Data Catalog（Amundsen / DataHub） | Theme 的 `where.data_sources_used` + 导航路径 | 数据发现、血缘、文档化 |
| Access Policies（RBAC/ABAC） | Theme 的 `who` + `where.access_boundary` | 权限控制、审计、合规 |
| Virtual Data Views | Theme 的 `how.join_logic` | 预定义关联逻辑，避免重复 SQL 构造 |

> **关键设计**：Semantic Layer 是**人的治理产物**，而 Data Theme 是**agent 使用经验的沉淀**。二者互补：Semantic Layer 提供“官方口径”，Data Theme 提供“实践路径”——agent 先用 Theme 快速找到数据，再用 Semantic Layer 验证口径。

## 4. 三阶记忆与既有形式化框架的整合

### 4.1 与 C5 三元组的映射

三阶记忆子架构记为 ℳ = ⟨Raw, Index, Theme⟩，经 Harness 契约 H 接入三层栈：

```
H: ℐ ──consult Theme──→ plan ──Index──→ locate facts ──Raw──→ ℰ
```

- **Raw ↔ D₁**：Tier 1 是 D₁ 的物理存储基底。agent 的 on-policy 取数最终落地到 Raw Data 层的具体切片。
- **Index ↔ D₂**：Tier 2 是 D₂ 的 off-policy 语义总结的具体实现。其产物不是无结构的 lake，而是 5W1H+Which 结构化 fact 库。
- **Theme ↔ D₃ + D₄**：Tier 3 同时承接 D₃（治理 memory）和 D₄（lifetime/时效策略）。Theme 的 `when` 维度直接编码 D₄ 的时新性与口径版本；Theme 的复用与收敛机制就是 D₃ 的“data-usage skill”实例化。

### 4.2 与控制面/数据面的正交切分（承接 [05-控制面与数据面正交切分](05-%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8E%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%AD%A3%E4%BA%A4%E5%88%87%E5%88%86.md)）

三阶记忆沿运行时 CP/DP 的投影：

| 层级 | 生产侧 | 消费侧 | 平面归属 |
|------|--------|--------|---------|
| **Tier 1 Raw** | 数据入湖、CDC、分区管理（后台运维） | on-policy 切片读取（D₁） | 生产侧：运维面；消费侧：运行时 DP |
| **Tier 2 Index** | OPIC 集群的 off-policy loop（独立进程） | semantic join 查询接口 | 生产侧：独立离线面；消费侧：运行时 DP |
| **Tier 3 Theme** | Theme 沉淀、匹配、演化（低频事件驱动） | Theme 查询、复用、继承 | 生产侧：运行时 CP；消费侧：运行时 CP |

> **关键观察**：Tier 3（Theme）是纯控制面——它决定“怎么取数”，但不搬运数据本身。这使 Theme 的复杂度增长（更多 Theme、更复杂的匹配逻辑）不会污染数据面 token 成本，直接支持 P4（平面分离）。

### 4.3 与 P7–P9 的增强

| 原命题 | 三阶记忆增强 | 新增可证伪判据 |
|--------|------------|---------------|
| **P7 取数解耦** | Raw 层规模增长（+src）被 Tier 2 Index 隔离，Theme 层进一步减少重复探索 | 固定 use case，Raw 规模 ×10 → 单请求 token 成本 ℓ 不变；且 Theme 命中率从 0% → 80% 时 ℓ 下降 |
| **P8 摘要充分性** | Tier 2 的 5W1H+Which fact 是 Σ 的 structured 形式；Tier 3 Theme 的复用证明 Σ 在 use case 上的充分性 | 三臂对比增加 (d) 5W1H+Which + Theme 复用：precision≈(b), coverage≈(a), token 成本 < (c) |
| **P9 治理收敛** | Theme 的继承/演化/废弃机制就是 π_data 的显式收敛路径 | 重复同类 use case n 次，测 Theme 命中率和决策熵 ℋ；期望命中率↑、熵↓、且收敛到稳定 Theme 版本 |

## 5. 与相关工作的划界（Novelty 定位）

### 5.1 既有工作覆盖了什么

| 维度 | 相关工作 | 它做了什么 | 它没做什么（= 本架构 delta） |
|------|---------|-----------|--------------------------|
| 个人知识管理 | Karpathy LLM Wiki (2026) | `raw/` + `wiki/` + `index.md` 的局部-first 知识库 | 无企业级治理、无 5W1H 本体、无多 agent 共享、无 Lakehouse 对接 |
| 事件本体 | XPEventCore (Piryani et al., 2024) | 5W1H 事件知识图谱 + EvCBR 推理 | 仅针对“事件”领域，无企业数据表、无 agent 推理轨迹、无 Which 维度、无 Theme 复用 |
| 语义索引 | SoKG (2601.10003v2) | 5W1H-guided QA 驱动 KG 构建 | 从文档构建静态 KG，无时效管理、无 Lakeshouse 集成、无 use-case 经验沉淀 |
| 语义层 | Enterprise Lakehouse Semantic Layer (Dremio / dbt / Cube) | 指标定义、虚拟视图、数据目录 | 面向**人**的 BI/分析，不面向**agent**的自主取数；无 agent 使用经验的记忆与复用 |
| 记忆架构 | MemGPT / Letta / Zep | 三层记忆（core/recall/archival）、时序知识图谱 | 面向**agent 自身交互**的记忆，不面向**外部企业数据**的理解与治理；无 5W1H 本体 |
| 数据治理 | DataGovBench / Five-Plane / DPDP | 访问控制、合规、审计 enforcement | 仅“管边界”，不把“agent 如何用数据”沉淀为可复用 skill / Theme |

### 5.2 四重 Novelty（数据层 v2.0 的独立贡献）

**N1 · 5W1H+Which 企业数据事实本体**：将新闻学/情报学的 5W1H 框架扩展为“Which”关联维度，并泛化到一切企业数据单元（人物、事件、报告、代码、表、工具调用），形成统一的知识原子。这是对 XPEventCore 和 SoKG 的 enterprise 扩展。

**N2 · 自底向上（Index）与自顶向下（Theme）的双螺旋记忆**：Index 是数据驱动（data tells you what it is），Theme 是任务驱动（organization tells data what it means）。二者闭环：Index 为 Theme 提供检索基底，Theme 为 Index 提供使用反馈（哪些 fact 被频繁引用 → 提升索引优先级）。这是 MemGPT/Letta 的“agent 自身记忆”与 Lakehouse “企业数据语义层”的**首次缝合**。

**N3 · Theme 作为 data-usage skill 的实例化与收敛机制**：把 D₃ 的抽象“data-usage skill”固化为可查询、可继承、可版本化的 Theme 记录，使 agent 的取数策略从“探索”收敛到“利用”。这是 agent 记忆理论（storage→reflection→experience）在**外部数据维度**的首次系统化实现。

**N4 · 三阶记忆与运行时 CP/DP 的第三次正交切分**：Raw↔DP（生产侧运维）、Index↔独立离线面、Theme↔CP（纯决策）。使 §05 的平面分离在数据子系统内再次成立，构成架构一致性的强证据。

## 6. 实验设计与可证伪命题

### 6.1 新增命题 P10–P12（三阶记忆专项）

**命题 P10（索引充分性 / Index Sufficiency）**

> **Tier 2 的 5W1H+Which 结构化 fact 对典型企业 use case 的覆盖度不低于直接扫描原始数据，且精度显著高于无结构化摘要。**

可证伪：四臂对比——(a) 裸扫描 Raw Data；(b) 人工 schema/元数据；(c) 无结构语义摘要（原 D₂ Σ）；(d) 5W1H+Which 结构化 fact。测 use case 上的 precision/recall/last-mile 成功率与 token 成本。P10 预期 (d) 在 precision 接近 (b)、coverage 接近 (a)、token 成本低于 (c)。

**命题 P11（Theme 收敛 / Theme Convergence）**

> **随同类 use case 重复运行，Data Theme 的匹配命中率单调上升，取数决策熵 ℋ 单调下降，且收敛到稳定 Theme 版本。**

可证伪：重复运行同类 use case n=1..50 次，记录每次 Theme 命中/创建/演化次数，测决策熵与成本方差。若命中率先升后降或发散 → Theme 匹配机制失效。

**命题 P12（双螺旋增益 / Double-Helix Gain）**

> **Index 与 Theme 的协同使用（Index 提供检索、Theme 提供策略）优于单独使用任一者。**

可证伪：三臂对比——(a) 仅用 Raw + D₁；(b) Raw + Index（无 Theme）；(c) Raw + Index + Theme。测复杂多步骤 use case 的成功率、token 成本、耗时。P12 预期 (c) > (b) > (a)。

### 6.2 实验平台建议

- **企业数据**：合成 ERP（SAP-style）+ 公开 TPC-DS / EasyTUS 基准表
- **SaaS 数据**：电商时间序列（模拟 Amazon 卖家数据）
- **公开数据**：arXiv 元数据 + 企业财报（模拟报告类 Raw Data）
- **agent 轨迹**：模拟多步骤金融分析 agent 的 reasoning chain，作为 Tier 1 的“agent 思考 memory”

## 7. 风险与待验证点

- [ ] **OPIC 集群的成本与时效**：大规模企业数据下，离线索引构建的 LLM 调用成本是否可控？摘要再生触发阈值如何设定？
- [ ] **5W1H+Which 的 schema 严格性**：不同数据类型的 5W1H 提取质量差异大（RDBMS 表易提取，非结构化报告难提取），是否需要 per-type 的提取模板？
- [ ] **Theme 匹配的歧义性**：同类业务问题在不同上下文中可能需要不同 Theme，如何避免“错误匹配”导致的取数偏差？
- [ ] **与现有企业数据治理体系的冲突**：Data Theme 可能与企业已有的 Semantic Layer / Data Catalog 产生口径冲突，需定义“Theme 口径 ← 服从 ← Semantic Layer 官方口径”的层级关系。
- [ ] **概率性控制面风险延伸**：Theme 的匹配与演化由 LLM 驱动，存在策略学偏风险 → 需可审计日志 + 不变量（Theme 不可授权越界，呼应 P3）。
